LESSON 01

스킬이란 무엇인가?

⏱ 예상 소요 시간: 15분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • Claude Code 스킬의 정의와 동작 원리를 설명할 수 있다
  • 스킬이 저장되는 위치(개인 vs 프로젝트 디렉토리)를 구분할 수 있다
  • 스킬, CLAUDE.md, 슬래시 명령어의 차이를 구분할 수 있다
  • 스킬이 적합한 커스터마이징 도구인 시나리오를 파악할 수 있다
핵심 요약
  • 스킬은 지시 사항이 담긴 폴더입니다. Claude Code가 더 정확하게 특정 작업을 수행하도록 탐색하고 활용할 수 있습니다. 각 스킬은 프론트매터(frontmatter)에 이름과 설명이 담긴 SKILL.md 파일로 이루어집니다.
  • Claude는 description을 보고 스킬을 매칭합니다. 작업을 요청하면, Claude는 사용 가능한 스킬 설명과 요청을 비교해 가장 잘 맞는 스킬을 활성화합니다.
  • 개인 스킬은 ~/.claude/skills에 저장되어 모든 프로젝트에서 사용됩니다.
  • 프로젝트 스킬은 레포지토리 내 .claude/skills에 저장되어 팀 전체와 공유됩니다.
  • 스킬은 필요할 때만 로드됩니다. CLAUDE.md(항상 로드)나 슬래시 명령어(명시적 호출)와 달리, 스킬은 Claude가 상황을 인식할 때 자동으로 활성화됩니다.
  • Claude에게 같은 내용을 반복해서 설명하고 있다면, 그건 스킬로 만들어야 할 신호입니다.

스킬이란 무엇인가?

매번 팀의 코딩 기준을 Claude에게 설명하고 계신가요? PR 리뷰마다 피드백 구조를 다시 설명하고, 커밋 메시지마다 원하는 형식을 상기시켜야 하나요? 스킬은 이 문제를 해결합니다.

스킬은 Claude에게 무언가를 한 번 가르치는 마크다운 파일입니다. 이후 Claude는 관련 상황이 발생할 때마다 자동으로 그 지식을 적용합니다.

스킬의 구조

스킬은 Claude Code가 더 정확하게 특정 작업을 처리할 수 있도록 탐색하고 활용하는 지시 사항과 리소스 폴더입니다. 각 스킬은 프론트매터에 이름과 설명이 담긴 SKILL.md 파일로 이루어집니다.

설명(description)은 Claude가 스킬을 사용할지 결정하는 핵심 기준입니다. PR 리뷰를 요청하면, Claude는 해당 요청을 사용 가능한 모든 스킬 설명과 비교해 적합한 스킬을 찾아 활성화합니다.

---
name: pr-review
description: PR의 코드 품질을 검토합니다. PR을 리뷰하거나 코드 변경사항을 확인할 때 사용합니다.
---

# 프론트매터 아래에 실제 지시 사항을 작성합니다
# 리뷰 체크리스트, 형식 규칙 등을 자유롭게 추가하세요

스킬 저장 위치

개인 스킬
~/.claude/skills
모든 프로젝트에서 사용됨. 커밋 스타일, 문서 형식 등 개인 취향.
프로젝트 스킬
.claude/skills
레포를 클론한 모든 팀원이 자동으로 공유. 팀 코딩 표준 등.
💡 Windows 사용자
개인 스킬은 C:/Users/<사용자명>/.claude/skills에 저장됩니다. 프로젝트 스킬은 버전 컨트롤에 커밋되어 팀 전체가 공유합니다.

스킬 vs CLAUDE.md vs 슬래시 명령어

기능 로드 방식 적합한 사용 사례
CLAUDE.md 매 대화마다 항상 로드 TypeScript strict mode처럼 항상 적용해야 하는 프로젝트 기준
스킬 요청과 매칭될 때 자동 로드 PR 리뷰, 커밋 메시지 등 특정 작업에만 필요한 전문 지식
슬래시 명령어 사용자가 명시적으로 입력 항상 직접 호출하고 싶은 기능

스킬 활용 시나리오

  • 팀이 따르는 코드 리뷰 기준
  • 선호하는 커밋 메시지 형식
  • 조직의 브랜드 가이드라인
  • 특정 유형의 문서 템플릿
  • 특정 프레임워크 디버깅 체크리스트
황금 법칙
Claude에게 같은 내용을 반복해서 설명하고 있다면, 그건 스킬로 만들어야 할 신호입니다.
레슨 돌아보기
  • 최근 Claude Code와 대화하면서 반복한 지시 사항은 무엇인가? 스킬로 만들었다면 얼마나 시간을 절약했을까?
  • 팀 워크플로에서 스킬로 인코딩하면 가장 도움될 기준이나 프로세스는 무엇인가?
LESSON 02

첫 번째 스킬 만들기

⏱ 예상 소요 시간: 20분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 올바른 프론트매터 구조로 스킬을 처음부터 만들 수 있다
  • Claude Code에서 스킬이 올바르게 로드되는지 테스트하고 확인할 수 있다
  • Claude Code가 들어오는 요청과 스킬을 매칭하는 방식을 설명할 수 있다
  • 스킬 우선순위 계층 구조를 설명할 수 있다
핵심 요약
  • 스킬은 프론트매터에 메타데이터(이름, 설명)가 담긴 SKILL.md 파일이 있는 디렉토리입니다.
  • Claude는 시작 시 스킬 이름과 설명만 로드하고, 들어오는 요청과 설명을 시맨틱 매칭으로 비교합니다.
  • 전체 스킬 콘텐츠를 컨텍스트에 로드하기 전에 확인 프롬프트가 표시됩니다.
  • 이름 충돌 시 우선순위: Enterprise → 개인 → 프로젝트 → 플러그인
  • 스킬을 업데이트하려면 SKILL.md를 편집하고, 제거하려면 디렉토리를 삭제하세요. 변경 후에는 항상 Claude Code를 재시작하세요.

스킬 만들기

모든 프로젝트에서 사용할 PR 설명 작성 스킬을 만들어 보겠습니다. 개인 스킬이므로 홈 디렉토리에 저장됩니다.

1단계: 스킬 폴더 생성

mkdir -p ~/.claude/skills/pr-description

2단계: SKILL.md 파일 생성

---
name: pr-description
description: 풀 리퀘스트 설명을 작성합니다. PR 생성, PR 작성, 또는 풀 리퀘스트의 변경사항 요약이 필요할 때 사용합니다.
---

PR 설명 작성 시:

1. `git diff main...HEAD`를 실행해 브랜치의 모든 변경사항 확인
2. 아래 형식으로 설명 작성:

## What (무엇을)
이 PR이 하는 일을 한 문장으로 설명

## Why (왜)
이 변경이 필요한 이유에 대한 간단한 맥락

## Changes (변경사항)
- 구체적인 변경사항을 불릿 포인트로
- 관련 변경사항끼리 묶어서 정리
- 삭제되거나 이름 변경된 파일 언급

스킬 테스트하기

Claude Code는 시작 시 스킬을 로드하므로, 스킬을 만든 후 세션을 재시작하세요. 그런 다음 사용 가능한 스킬 목록을 확인해 제대로 로드됐는지 확인할 수 있습니다.

브랜치에서 변경사항을 만든 뒤 "내 변경사항에 대한 PR 설명을 작성해줘"라고 요청해 보세요. Claude가 PR 설명 스킬을 사용하고 있다고 알리며, diff를 확인한 후 매번 동일한 형식으로 설명을 작성할 것입니다.

스킬 매칭 원리

Claude Code가 시작되면 네 곳에서 스킬을 스캔하지만, 이름과 설명만 로드합니다 (전체 내용은 아님). 이것이 중요한 포인트입니다.

요청을 보내면 Claude는 사용 가능한 모든 스킬의 설명과 메시지를 비교합니다. 예를 들어 "이 함수가 하는 일을 설명해줘"는 "시각적 다이어그램으로 코드를 설명"이라는 설명을 가진 스킬과 매칭될 수 있는데, 의도가 겹치기 때문입니다.

매칭 후 확인 단계
스킬이 매칭되면 Claude는 스킬을 로드할지 확인을 요청합니다. 이 확인 단계는 Claude가 어떤 컨텍스트를 불러오는지 인지할 수 있도록 도와줍니다. 확인 후 Claude는 완전한 SKILL.md 파일을 읽고 지시사항을 따릅니다.

스킬 우선순위

같은 이름의 스킬이 여러 위치에 있을 때 우선순위 순서:

1순위 (최고)
Enterprise
관리자가 배포한 조직 설정
2순위
개인 (Personal)
~/.claude/skills
3순위
프로젝트 (Project)
레포지토리 내 .claude/skills
4순위 (최저)
플러그인 (Plugins)
설치된 플러그인
⚠️ 주의
충돌을 방지하려면 구체적인 이름을 사용하세요. "review" 대신 "frontend-review" 또는 "backend-review"처럼 명확하게 작성하세요.

스킬 업데이트 및 제거

  • 스킬 업데이트: SKILL.md 파일 편집
  • 스킬 제거: 스킬 디렉토리 삭제
  • 변경 적용: Claude Code 재시작 필수
레슨 돌아보기
  • 지금 당장 스킬로 만들 수 있는 일상 워크플로우 작업은 무엇인가? 설명은 어떻게 작성할까?
  • 우선순위 계층이 팀의 스킬 관리 전략에 어떤 영향을 미칠까? 개인 스킬과 프로젝트 스킬 중 어느 것에 더 의존할 것인가?
LESSON 03

설정 및 멀티파일 스킬

⏱ 예상 소요 시간: 20분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • allowed-tools, model 등 고급 스킬 메타데이터 필드를 설정할 수 있다
  • 올바른 요청에 안정적으로 트리거되는 효과적인 스킬 설명을 작성할 수 있다
  • allowed-tools를 사용해 스킬 활성화 시 Claude의 도구 접근을 제한할 수 있다
  • 점진적 공개(progressive disclosure)와 멀티파일 구조로 복잡한 스킬을 구성할 수 있다
핵심 요약
  • name과 description은 필수 — allowed-tools와 model은 선택이지만 강력한 추가 기능입니다.
  • 좋은 설명은 두 가지 질문에 답합니다: 스킬이 무엇을 하는가? Claude가 언제 사용해야 하는가?
  • allowed-tools는 스킬이 활성화될 때 Claude가 사용할 수 있는 도구를 제한합니다. 읽기 전용 또는 보안 민감 워크플로우에 유용합니다.
  • 점진적 공개: SKILL.md는 500줄 미만으로 유지하고, 필요할 때만 읽는 보조 파일(참고자료, 스크립트, 에셋)에 링크합니다.
  • 스크립트는 내용을 컨텍스트에 로드하지 않고 실행됩니다 — 출력만 토큰을 소비하므로 컨텍스트를 효율적으로 유지합니다.

스킬 메타데이터 필드

필드필수 여부설명
name ✅ 필수 스킬 식별자. 소문자, 숫자, 하이픈만 사용. 최대 64자. 디렉토리명과 일치해야 함.
description ✅ 필수 Claude가 언제 스킬을 사용할지 알려줌. 최대 1,024자. 매칭에 가장 중요한 필드.
allowed-tools ⬜ 선택 스킬 활성화 시 Claude가 사용할 수 있는 도구를 제한합니다.
model ⬜ 선택 스킬에 사용할 Claude 모델을 지정합니다.

효과적인 설명 작성법

지시사항을 명확하게 작성하세요. "문서 작성을 도와라"처럼 모호한 설명은 Claude가 무엇을 해야 할지 알 수 없습니다. 좋은 설명은 두 가지 질문에 답합니다:

  • 이 스킬이 무엇을 하는가?
  • Claude가 언제 이 스킬을 사용해야 하는가?
트리거 키워드 추가
스킬이 예상대로 트리거되지 않으면, 실제로 사용하는 표현과 일치하는 키워드를 설명에 더 추가하세요. 설명이 Claude의 매칭 기준이 되기 때문에 언어 선택이 중요합니다.

allowed-tools로 도구 제한하기

읽기만 가능하고 수정은 불가능한 스킬을 원할 때, 보안 민감 워크플로우나 읽기 전용 작업에 유용합니다.

---
name: codebase-onboarding
description: 새 개발자가 시스템 구조를 이해하도록 돕습니다.
allowed-tools: Read, Grep, Glob, Bash
model: sonnet
---

allowed-tools를 생략하면 스킬은 아무것도 제한하지 않으며, Claude는 일반 권한 모델을 사용합니다.

점진적 공개 (Progressive Disclosure)

스킬은 Claude의 컨텍스트 창을 대화와 공유합니다. 2,000줄짜리 단일 파일은 두 가지 문제가 있습니다: 컨텍스트 창을 많이 차지하고 유지관리가 어렵습니다.

점진적 공개로 해결하세요. SKILL.md에는 핵심 지시사항만 유지하고, 필요할 때만 읽는 별도의 참고자료 파일에 세부 내용을 넣습니다.

my-skill/
  ├── SKILL.md         # 핵심 지시사항 (500줄 미만 권장)
  ├── scripts/        # 실행 코드
  ├── references/     # 추가 문서
  └── assets/         # 이미지, 템플릿, 데이터 파일
황금 법칙
SKILL.md는 500줄 미만으로 유지하세요. 그 이상이라면 내용을 별도의 참고 파일로 분리하는 것을 고려하세요.

스크립트 효율적으로 활용하기

스킬 디렉토리의 스크립트는 내용을 컨텍스트에 로드하지 않고 실행할 수 있습니다. 스크립트가 실행되면 출력만 토큰을 소비합니다. SKILL.md에 포함해야 할 핵심 지시사항은 Claude에게 스크립트를 읽으라가 아닌 실행하라고 알리는 것입니다.

특히 유용한 경우:

  • 환경 검증
  • 일관성이 필요한 데이터 변환
  • 생성 코드보다 테스트된 코드로 처리하는 것이 더 신뢰할 수 있는 작업
레슨 돌아보기
  • 여러 파일이 필요한 스킬을 만들려면 SKILL.md와 보조 참고 파일을 어떻게 구성할까?
  • 팀 워크플로우 중 allowed-tools로 도구를 제한하면 중요한 안전 레이어가 될 수 있는 부분이 있을까?
LESSON 04

스킬 vs 다른 Claude Code 기능들

⏱ 예상 소요 시간: 15분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 스킬을 CLAUDE.md, 서브에이전트, 훅, MCP 서버와 비교할 수 있다
  • 특정 사용 사례에 맞는 Claude Code 커스터마이징 기능을 선택할 수 있다
  • 여러 기능을 효과적으로 조합하는 구성을 설계할 수 있다
핵심 요약
  • CLAUDE.md는 매 대화마다 로드되어 항상 켜져 있는 프로젝트 기준에 적합합니다. 스킬은 요청에 따라 로드되어 특정 작업 전문 지식에 적합합니다.
  • 서브에이전트는 격리된 실행 컨텍스트에서 실행됩니다. 위임 작업에 사용하세요. 스킬은 현재 대화에 지식을 추가합니다.
  • 훅은 이벤트 기반(파일 저장, 도구 호출 시 실행)입니다. 스킬은 요청 기반(요청 내용에 따라 활성화)입니다.
  • MCP 서버는 외부 도구와 통합을 제공합니다 — 스킬과는 전혀 다른 범주입니다.
  • 각 기능은 고유한 역할이 있습니다 — 모든 것을 한 기능에 억지로 넣지 말고 조합해서 사용하세요.

CLAUDE.md vs 스킬

기준CLAUDE.md스킬
로드 시점 매 대화마다 항상 요청에 매칭될 때만
적합한 사용 사례 TypeScript strict mode, "절대 DB 스키마 수정 금지" PR 리뷰 체크리스트, 커밋 메시지 형식
컨텍스트 소비 항상 컨텍스트 소비 필요할 때만 소비

스킬 vs 서브에이전트

스킬은 현재 대화에 지식을 추가합니다. 스킬이 활성화되면 지시사항이 기존 컨텍스트에 합류합니다.

서브에이전트는 별도의 컨텍스트에서 실행됩니다. 작업을 받고 독립적으로 처리한 후 결과를 반환합니다. 메인 대화와 격리되어 있습니다.

스킬 사용 시나리오
지식 강화
현재 작업에 대한 전문 지식을 추가할 때, 대화 전반에 걸쳐 전문성이 적용될 때
서브에이전트 사용 시나리오
격리 실행
별도 실행 컨텍스트에서 작업을 위임할 때, 메인 컨텍스트와 격리가 필요할 때

스킬 vs 훅

훅은 이벤트에 반응합니다. Claude가 파일을 저장할 때마다 린터를 실행하거나, 특정 도구 호출 전에 입력을 검증할 수 있습니다. 이벤트 기반입니다.

스킬은 요청 기반입니다. 사용자가 요청하는 내용에 따라 활성화됩니다.

전체적인 조합

일반적인 설정은 각 기능의 강점을 활용해 조합합니다:

기능역할
CLAUDE.md항상 켜져 있는 프로젝트 기준
스킬필요할 때 로드되는 특정 작업 전문 지식
훅이벤트로 트리거되는 자동화 작업
서브에이전트위임된 작업을 위한 격리 실행 컨텍스트
MCP 서버외부 도구 및 통합
핵심 원칙
더 적합한 옵션이 있는데 모든 것을 스킬에 억지로 넣지 마세요. 여러 기능을 동시에 사용해도 됩니다. 스킬은 자동으로 관련 지식을 제공하고, CLAUDE.md는 항상 켜진 지시사항, 서브에이전트는 격리 실행, 훅은 이벤트 처리, MCP는 외부 도구를 담당합니다.
레슨 돌아보기
  • 현재 CLAUDE.md 파일을 살펴보세요. 스킬로 더 잘 작동할 내용(가끔만 필요한 지식)이 있나요?
  • 팀의 개발 워크플로우에서 어떤 Claude Code 기능 조합(스킬, 훅, 서브에이전트, MCP)이 가장 큰 문제를 해결할까요?
LESSON 05

스킬 공유하기

⏱ 예상 소요 시간: 20분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • Git 레포지토리에 커밋해 팀과 스킬을 공유할 수 있다
  • 플러그인과 마켓플레이스를 통해 프로젝트 전반에 스킬을 배포할 수 있다
  • Enterprise 관리 설정으로 조직 전체에 스킬을 배포할 수 있다
  • 특정 스킬을 사용하는 커스텀 서브에이전트를 구성할 수 있다
핵심 요약
  • 프로젝트 스킬은 .claude/skills에 있어 Git을 통해 자동 공유됩니다 — 레포를 클론하는 모든 사람이 자동으로 받습니다.
  • 플러그인으로 마켓플레이스를 통해 레포지토리 전반에 스킬을 배포할 수 있습니다.
  • Enterprise 관리 설정으로 조직 전체에 스킬을 최고 우선순위로 배포합니다. 필수 기준과 컴플라이언스에 이상적입니다.
  • 서브에이전트는 자동으로 스킬을 보지 못합니다 — 커스텀 에이전트의 프론트매터 skills 필드에 명시적으로 나열해야 합니다.
  • 내장 에이전트(Explorer, Plan, Verify)는 스킬에 전혀 접근할 수 없습니다 — .claude/agents에 정의된 커스텀 서브에이전트만 가능합니다.

레포지토리에 스킬 커밋하기

가장 간단한 공유 방법입니다. .claude/skills에 스킬을 배치하면, 레포를 클론하는 모든 사람이 별도 설치 없이 자동으로 받습니다.

.claude/
  ├── agents/
  ├── hooks/
  ├── skills/       # 팀 스킬이 여기에
      └── pr-review/
          └── SKILL.md
  └── settings.json

업데이트를 푸시하면 다음 풀 때 모든 사람이 업데이트를 받습니다. 이 방법은 팀 코딩 기준, 프로젝트별 워크플로우, 코드베이스 구조를 참조하는 스킬에 적합합니다.

플러그인으로 스킬 배포하기

플러그인은 팀과 프로젝트 간에 공유되도록 설계된 커스텀 기능으로 Claude Code를 확장하는 방법입니다. 플러그인 프로젝트에서 skills 디렉토리를 만들고, 각 스킬에 SKILL.md가 있는 폴더를 생성합니다.

마켓플레이스에 플러그인을 배포하면, 다른 사용자들이 발견하고 Claude Code에 설치할 수 있습니다. 스킬이 프로젝트에 너무 특화되지 않고 더 넓은 커뮤니티에도 유용할 때 적합합니다.

Enterprise 관리 설정으로 배포

관리자는 관리 설정을 통해 조직 전체에 스킬을 배포할 수 있습니다. Enterprise 스킬은 가장 높은 우선순위를 가집니다 — 같은 이름의 개인, 프로젝트, 플러그인 스킬을 재정의합니다.

"strictKnownMarketplaces": [
  {
    "source": "github",
    "repo": "acme-corp/approved-plugins"
  },
  {
    "source": "npm",
    "package": "@acme-corp/compliance-plugins"
  }
]
⚠️ Enterprise 스킬 사용 기준
필수 기준, 보안 요구사항, 컴플라이언스 워크플로우, 조직 전체에서 반드시 일관해야 하는 코딩 관행에만 사용하세요.

스킬과 서브에이전트

중요한 사실: 서브에이전트는 자동으로 스킬을 보지 못합니다. 작업을 서브에이전트에 위임하면, 새롭고 깨끗한 컨텍스트로 시작합니다.

  • 내장 에이전트(Explorer, Plan, Verify)는 스킬에 전혀 접근할 수 없습니다.
  • 커스텀 서브에이전트는 스킬을 사용할 수 있지만, 명시적으로 나열해야 합니다.
  • 스킬은 메인 대화처럼 요청에 따라가 아닌, 서브에이전트 시작 시 로드됩니다.

커스텀 서브에이전트를 만들려면 .claude/agents에 에이전트 마크다운 파일을 추가하고 skills 필드를 사용합니다:

---
name: frontend-security-accessibility-reviewer
description: "프론트엔드 코드를 접근성과 보안 관점에서 리뷰할 때 사용합니다..."
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch, Skill...
model: sonnet
color: blue
skills: accessibility-audit, performance-check
---

  • 특정 스킬을 가진 커스텀 서브에이전트가 위임 작업의 일관성을 개선할 수 있는 워크플로우가 있는가?
LESSON 06

스킬 트러블슈팅

⏱ 예상 소요 시간: 15분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 스킬 검증 도구로 디버깅 전에 구조적 문제를 발견할 수 있다
  • 스킬 트리거 및 로딩 문제를 진단하고 해결할 수 있다
  • Enterprise, 개인, 프로젝트, 플러그인 스킬 간 우선순위 충돌을 해결할 수 있다
  • 누락된 의존성, 권한, 경로 문제 등 런타임 오류를 디버그할 수 있다
핵심 요약
  • 스킬 검증 도구로 시작하세요 — 다른 것을 디버깅하기 전에 구조적 문제를 잡아줍니다.
  • 스킬이 트리거되지 않으면, 원인은 거의 항상 description입니다 — 실제로 요청을 표현하는 방식과 일치하는 트리거 구문을 추가하세요.
  • 스킬이 로드되지 않으면, SKILL.md가 skills 루트가 아닌 이름 있는 디렉토리 안에 있는지, 파일명이 정확히 SKILL.md인지 확인하세요.
  • 잘못된 스킬이 사용되면, 설명이 너무 비슷한 겁니다 — 더 구별되게 만드세요.
  • 런타임 오류의 경우 의존성, 파일 권한(chmod +x), 경로 구분자(어디서든 슬래시 사용)를 확인하세요.

스킬 검증 도구 사용

가장 먼저 에이전트 스킬 검증 명령어를 사용하세요. 설치 방법은 운영체제마다 다르지만, uv를 사용하면 빠르게 설정할 수 있습니다.

설치 후 스킬 디렉토리로 이동하거나 어디서든 명령어를 실행하세요. 검증 도구는 다른 것을 디버깅하기 전에 구조적 문제를 잡아줍니다.

트러블슈팅 체크리스트

증상원인해결책
트리거되지 않음 description이 요청과 매칭되지 않음 설명 개선, 트리거 구문 추가, 다양한 표현으로 테스트
로드되지 않음 파일 구조 오류 이름 있는 디렉토리 안에 SKILL.md 위치, 파일명 정확히 확인
잘못된 스킬 사용 설명이 너무 유사 설명을 더 구별되게 수정
우선순위 충돌 더 높은 우선순위의 같은 이름 스킬 존재 스킬 이름 변경 또는 관리자에게 문의
플러그인 스킬 미표시 캐시 또는 구조 오류 캐시 지우기, 재시작, 재설치
런타임 오류 의존성/권한/경로 문제 패키지 설치, chmod +x, 슬래시 통일

스킬이 트리거되지 않을 때

스킬이 존재하고 검증을 통과했는데, Claude가 예상대로 사용하지 않으면 — 원인은 거의 항상 description입니다.

Claude는 시맨틱 매칭을 사용하므로, 요청이 설명의 의미와 겹쳐야 합니다. 겹침이 충분하지 않으면 매칭이 안 됩니다:

  • 실제로 요청하는 방식과 설명을 비교하세요
  • 사용자가 실제로 쓸 표현의 트리거 구문을 추가하세요
  • "이걸 프로파일링 도와줘", "왜 느리지?", "이걸 더 빠르게" 등 다양한 표현으로 테스트하세요
  • 어떤 표현이 트리거에 실패하면, 그 키워드를 설명에 추가하세요

스킬이 로드되지 않을 때

"사용 가능한 스킬 목록"에 스킬이 표시되지 않으면 구조적 요구사항을 확인하세요:

  • SKILL.md 파일이 skills 루트가 아닌 이름 있는 디렉토리 안에 있어야 합니다
  • 파일명이 정확히 SKILL.md여야 합니다 — "SKILL"은 대문자, "md"는 소문자
claude --debug  # 로딩 오류를 보려면 디버그 모드로 실행

런타임 오류

스킬이 로드되지만 실행 중 실패합니다. 일반적인 원인:

누락된 의존성
패키지 설치 필요
스킬이 외부 패키지를 사용하면 설치되어 있어야 합니다. 스킬 설명에 의존성 정보를 추가하세요.
권한 문제
chmod +x 필요
스크립트에 실행 권한이 필요합니다. 스킬이 참조하는 모든 스크립트에 chmod +x를 실행하세요.
경로 구분자
슬래시 통일
Windows에서도 어디서든 슬래시(/)를 사용하세요.
