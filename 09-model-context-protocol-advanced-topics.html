레슨 1 · Sampling

Sampling: 서버가 AI를 빌리는 방법

MCP 서버가 직접 Claude를 호출하지 않고, 클라이언트를 통해 LLM 기능을 위임하는 혁신적인 패턴을 학습합니다.

🎯 학습 목표

  • Sampling이 해결하는 문제와 그 작동 원리를 설명할 수 있다
  • 서버 측과 클라이언트 측의 Sampling 구현 방법을 이해한다
  • 언제 Sampling을 활용해야 하는지 판단할 수 있다

💡 핵심 요약

  • Sampling은 MCP 서버가 클라이언트에게 "Claude를 대신 호출해 달라"고 요청하는 메커니즘
  • 서버는 API 키 없이도 AI 기능을 활용할 수 있어 복잡성과 비용이 크게 감소
  • 공개 MCP 서버에 특히 유용 — 서버가 아닌 각 클라이언트가 AI 사용 비용을 부담
  • 서버: ctx.session.create_message() 호출 / 클라이언트: sampling_callback 구현

Sampling이 해결하는 문제

위키피디아에서 정보를 수집해 보고서로 요약하는 MCP 서버를 만들 때 어떻게 해야 할까요? 데이터를 가져왔지만, 이제 AI로 요약해야 합니다.

방법 서버에 직접 Claude 통합 Sampling 활용
API 키 필요 서버에 직접 API 키 보관 필요 클라이언트가 이미 보유, 서버는 불필요
비용 부담 서버 운영자가 모든 AI 비용 부담 각 클라이언트가 자신의 AI 비용 부담
구현 복잡성 높음 — 인증, 오류 처리 등 직접 구현 낮음 — 단순 요청 전달
공개 서버 적합성 불리 — 무제한 AI 비용 발생 위험 최적 — 사용자별 비용 독립

Sampling 작동 흐름

1

서버 작업 완료

서버가 도구 함수 내에서 본래 작업(예: 위키피디아 검색)을 완료합니다.

↓
2

프롬프트 생성 및 Sampling 요청

서버가 텍스트 생성이 필요한 프롬프트를 구성하고 ctx.session.create_message()로 클라이언트에 요청합니다.

↓
3

클라이언트가 Claude 호출

클라이언트의 sampling_callback이 호출되어 Anthropic SDK로 Claude에게 요청합니다.

↓
4

결과 반환

생성된 텍스트가 클라이언트 → 서버로 전달되고, 서버는 이를 최종 응답에 활용합니다.

서버 측 구현

서버의 도구 함수 내에서 ctx.session.create_message()를 사용합니다:

@mcp.tool() async def summarize(text_to_summarize: str, ctx: Context): prompt = f""" Please summarize the following text: {text_to_summarize} """ result = await ctx.session.create_message( messages=[ SamplingMessage( role="user", content=TextContent( type="text", text=prompt ) ) ], max_tokens=4000, system_prompt="You are a helpful research assistant", ) if result.content.type == "text": return result.content.text else: raise ValueError("Sampling failed")

클라이언트 측 구현

클라이언트는 sampling_callback을 구현하고 세션 초기화 시 전달합니다:

async def sampling_callback( context: RequestContext, params: CreateMessageRequestParams ): # Anthropic SDK로 Claude 호출 text = await chat(params.messages) return CreateMessageResult( role="assistant", model=model, content=TextContent(type="text", text=text), ) # 세션 초기화 시 콜백 전달 async with ClientSession( read, write, sampling_callback=sampling_callback # ← 핵심 ) as session: await session.initialize()
💡

언제 Sampling을 써야 할까?

공개 접근 가능한 MCP 서버를 구축할 때가 핵심 사용 사례입니다. 불특정 다수 사용자가 AI를 무제한으로 사용하면서 그 비용이 서버 운영자에게 전가되는 것을 방지할 수 있습니다.

  • 클라이언트가 sampling_callback을 구현하지 않은 경우, 서버의 Sampling 요청은 어떻게 처리될까요?
  • Sampling을 이용한 요청에서 클라이언트는 서버의 프롬프트를 수정할 수 있을까요? 이것이 보안적으로 어떤 의미를 가질까요?
레슨 2 · Sampling Walkthrough

Sampling 단계별 구현 가이드

실제 프로젝트를 통해 Sampling의 모든 구현 단계를 체계적으로 살펴봅니다.

🎯 학습 목표

  • 서버에서 create_message()를 올바르게 초기화하는 방법을 익힌다
  • MCP 메시지 형식과 Anthropic SDK 형식의 차이를 이해하고 변환한다
  • Sampling 콜백의 전체 생명주기를 파악한다

💡 핵심 요약

  • 서버는 create_message()로 Sampling을 시작하며, 메시지는 MCP 형식으로 전달됨
  • MCP 메시지와 Anthropic SDK 메시지는 형식이 달라 변환 로직이 필요
  • 클라이언트는 생성된 텍스트를 CreateMessageResult로 감싸 반환
  • 콜백은 반드시 ClientSession 초기화 시 전달해야 함 — 나중에 추가 불가
  • 생성된 텍스트 수신 후 서버는 워크플로우에 활용하거나 재호출하거나 그대로 반환 가능

구현 단계 상세 분석

1

Sampling 시작 (서버)

도구 호출 중 create_message() 메서드를 실행합니다. LLM에 전송할 메시지 목록을 포함시킵니다.

↓
2

Sampling 콜백 (클라이언트)

클라이언트는 반드시 sampling callback을 구현해야 합니다. 이 콜백은 서버가 제공한 메시지 목록을 받습니다.

↓
3

메시지 형식 변환

서버가 전달한 MCP 형식 메시지는 사용 중인 LLM SDK와 호환되지 않을 수 있습니다. Anthropic SDK의 경우 변환 로직을 작성해야 합니다.

↓
4

생성 텍스트 반환

LLM으로 텍스트를 생성한 후 CreateMessageResult에 담아 반환합니다. 이 결과가 서버로 전달됩니다.

↓
5

콜백 연결

클라이언트의 콜백을 ClientSession 호출에 전달하는 것을 잊지 마세요.

↓
6

결과 활용 (서버)

서버에서 생성된 텍스트를 받은 후: 워크플로우에 사용, 추가 Sampling 요청, 또는 그대로 반환할 수 있습니다.

server.py 핵심 코드

mcp = FastMCP(name="DemoServer") @mcp.tool() async def summarize(text_to_summarize: str, ctx: Context): prompt = f"""Please summarize the following text: {text_to_summarize}""" result = await ctx.session.create_message( messages=[ SamplingMessage( role="user", content=TextContent(type="text", text=prompt) ) ], max_tokens=4000, system_prompt="You are a helpful research assistant", ) if result.content.type == "text": return result.content.text # 생성된 텍스트 반환
⚠️

메시지 형식 호환성 주의

MCP 메시지 형식은 모든 LLM SDK와 직접 호환되지 않습니다. Anthropic SDK 사용 시 MCP SamplingMessage를 SDK 형식으로 변환하는 코드가 필요합니다. 이 변환 로직을 미리 준비해 두세요.

레슨 3 · Notifications

로그와 진행률 알림으로 UX 개선

장시간 실행되는 도구의 상태를 실시간으로 클라이언트에 전달해 답답한 대기 경험을 해소합니다.

🎯 학습 목표

  • context.info()와 context.report_progress()로 알림을 구현할 수 있다
  • 클라이언트에서 로그 콜백과 진행률 콜백을 설정하는 방법을 이해한다
  • 알림이 다양한 애플리케이션 유형에서 어떻게 표시될 수 있는지 파악한다

💡 핵심 요약

  • 알림은 선택 사항이지만, 사용자 경험에 엄청난 차이를 만들어냄
  • context.info(), context.warning(), context.debug(), context.error()로 다양한 레벨의 로그 전송
  • context.report_progress(progress, total)로 진행률 업데이트 전송
  • 로그 콜백은 ClientSession 생성 시, 진행률 콜백은 call_tool() 호출 시 전달
  • 알림 표시 방식은 클라이언트 애플리케이션이 결정 — CLI, 웹, 데스크탑 각각 다름

왜 알림이 중요한가?

Claude가 오래 걸리는 도구(예: 리서치, 데이터 처리)를 호출할 때 기본적으로 사용자는 작업이 완료될 때까지 아무것도 볼 수 없습니다. 마치 프로그램이 멈춘 것처럼 보이죠. 로그와 진행률 알림을 활성화하면 사용자는 뒤에서 무슨 일이 일어나는지 실시간으로 볼 수 있습니다.

서버 측: Context 객체 활용

@mcp.tool( name="research", description="Research a given topic" ) async def research( topic: str = Field(description="Topic to research"), *, context: Context ): await context.info("About to do research...") # 로그 전송 await context.report_progress(20, 100) # 진행률 20% sources = await do_research(topic) await context.info("Writing report...") await context.report_progress(70, 100) # 진행률 70% results = await generate_report(sources) return results

클라이언트 측: 콜백 구현

async def logging_callback(params: LoggingMessageNotificationParams): print(params.data) # 로그 메시지 출력 async def print_progress_callback( progress: float, total: float | None, message: str | None ): if total is not None: percentage = (progress / total) * 100 print(f"Progress: {progress}/{total} ({percentage:.1f}%)") else: print(f"Progress: {progress}") async with ClientSession( read, write, logging_callback=logging_callback # ← 세션 생성 시 전달 ) as session: await session.initialize() await session.call_tool( name="research", arguments={"topic": "MCP"}, progress_callback=print_progress_callback # ← 툴 호출 시 전달 )

레슨 4 · Roots

Roots: 파일 시스템 접근 권한 관리

MCP 서버에 특정 파일과 폴더에 대한 접근 권한을 부여하면서도 보안 경계를 유지하는 방법을 학습합니다.

🎯 학습 목표

  • Roots가 사용자 경험과 보안 모두를 개선하는 원리를 설명할 수 있다
  • 클라이언트에서 Roots를 제공하고 서버에서 접근하는 방법을 구현한다
  • is_path_allowed() 패턴으로 파일 접근 권한을 검증할 수 있다

💡 핵심 요약

  • Roots = "이 파일/폴더들은 MCP 서버가 접근해도 됩니다"라고 알려주는 권한 시스템
  • 사용자가 전체 경로 없이 파일명만 말해도 Claude가 Roots에서 파일을 찾을 수 있음
  • MCP SDK는 자동으로 Roots 제한을 적용하지 않음 — 직접 검증 코드를 작성해야 함
  • 서버는 ctx.session.list_roots()로 허용된 경로 목록을 조회
  • URI는 반드시 file://로 시작해야 함 (MCP 스펙 요구사항)

Roots가 해결하는 문제

파일을 MP4에서 MOV로 변환하는 MCP 서버를 상상해 보세요. 사용자가 "biking.mp4를 변환해 줘"라고 하면:

  • Roots 없이: Claude는 파일이 어디 있는지 알 수 없어 전체 경로 입력을 요구함
  • Roots 있이: Claude가 허용된 디렉토리에서 파일을 검색해 자동으로 전체 경로를 파악

Roots 활용 워크플로우

1

"biking.mp4를 mov 형식으로 변환해 줘"

↓
3

파일 탐색

허용된 디렉토리에서 read_dir로 파일을 찾습니다.

↓
4

변환 도구 호출

전체 경로를 찾은 후 변환 도구를 호출합니다.

구현: 클라이언트 측

# CLI 인수로 허용할 경로 수신 root_paths = sys.argv[1:] # 예: /path/to/videos # Root 객체 생성 (file:// URI 형식) def create_roots(paths): return [Root(uri=f"file://{path}") for path in paths] # 루트 목록 콜백 async def list_roots_callback(): return ListRootsResult(roots=create_roots(root_paths)) # 세션 초기화 시 콜백 전달 async with ClientSession( read, write, list_roots_callback=list_roots_callback ) as session: await session.initialize()

구현: 서버 측 — 접근 권한 검증

# 서버에서 Roots 조회 async def get_approved_paths(ctx: Context): roots_result = await ctx.session.list_roots() return [root.uri.replace("file://", "") for root in roots_result.roots] # ⚠️ 반드시 직접 구현해야 함: MCP SDK는 자동으로 제한하지 않음! async def is_path_allowed(requested_path: str, ctx: Context) -> bool: approved = await get_approved_paths(ctx) return any(requested_path.startswith(root) for root in approved) # 모든 파일 접근 도구에서 검증 실행 @mcp.tool() async def convert_video(file_path: str, ctx: Context): if not await is_path_allowed(file_path, ctx): raise PermissionError(f"Access denied: {file_path}") # 실제 변환 로직...

SDK 자동 제한 없음 — 직접 구현 필수

MCP SDK는 서버가 접근하는 파일 경로를 자동으로 제한하지 않습니다. is_path_allowed() 같은 검증 함수를 파일/디렉토리에 접근하는 모든 도구에 직접 적용해야 합니다.

레슨 5 · JSON Message Types

MCP의 JSON 메시지 타입 이해

MCP 통신의 기반이 되는 JSON 메시지 타입을 분류하고, 양방향 프로토콜의 설계 원리를 이해합니다.

🎯 학습 목표

  • 요청-결과 메시지와 알림 메시지의 차이를 설명할 수 있다
  • 클라이언트 메시지와 서버 메시지를 구분한다
  • MCP가 양방향 프로토콜인 이유와 그 중요성을 이해한다

💡 핵심 요약

  • 모든 MCP 통신은 JSON 메시지를 통해 이루어짐
  • 메시지는 두 범주: 요청-결과 쌍(응답 필요)과 알림(단방향, 응답 불필요)
  • 서버와 클라이언트 모두 메시지를 시작할 수 있음 — MCP는 양방향 프로토콜
  • 공식 스펙은 GitHub의 별도 저장소에 TypeScript로 정의됨
  • 서버가 클라이언트에게 메시지를 보낼 수 있다는 점이 HTTP 트랜스포트 선택에 영향을 줌

메시지 범주

요청-결과 메시지 (Request-Result Pairs)

항상 쌍으로 옵니다. 요청을 보내면 결과를 기대합니다:

요청결과
Call Tool RequestCall Tool Result
List Prompts RequestList Prompts Result
Read Resource RequestRead Resource Result
Initialize RequestInitialize Result

알림 메시지 (Notifications)

단방향 이벤트 알림 — 응답 불필요:

📊 Progress Notification
📝 Logging Message Notification
🔧 Tool List Changed Notification
📄 Resource Updated Notification

클라이언트 vs 서버 메시지

발신자메시지 종류예시
클라이언트 → 서버 대부분의 요청 도구 호출, 리소스 읽기, 프롬프트 목록 요청
서버 → 클라이언트 Sampling, Roots 요청 + 알림 Create Message Request, List Roots Request, Progress Notification
💡

왜 이것이 중요한가?

서버가 클라이언트에게 메시지를 시작할 수 있다는 것이 트랜스포트 선택에 핵심적 영향을 줍니다. 일부 HTTP 설정에서는 서버가 클라이언트로 메시지를 보내기 어렵기 때문에, Sampling이나 진행률 알림 같은 기능이 제한될 수 있습니다.

레슨 6 · STDIO Transport

STDIO 트랜스포트: 가장 단순한 통신 방식

표준 입출력 스트림을 통해 클라이언트와 서버가 통신하는 가장 기본적이고 직관적인 MCP 트랜스포트를 이해합니다.

🎯 학습 목표

  • STDIO 트랜스포트의 작동 원리와 적합한 사용 시나리오를 설명할 수 있다
  • MCP 연결 초기화 시 필요한 3단계 핸드셰이크를 이해한다
  • STDIO와 HTTP 트랜스포트의 차이점을 파악한다

💡 핵심 요약

  • STDIO 트랜스포트: 클라이언트가 서버를 서브프로세스로 실행, stdin/stdout으로 통신
  • 클라이언트와 서버가 같은 머신에서 실행될 때만 사용 가능
  • 개발/테스트에 완벽 — 터미널에서 직접 JSON 메시지를 붙여넣어 테스트 가능
  • 모든 MCP 연결은 3단계 핸드셰이크로 시작: Initialize Request → Initialize Result → Initialized Notification
  • 양방향 통신이 자연스러움 — HTTP 트랜스포트의 제약을 이해하기 위한 기준점

STDIO 트랜스포트 작동 원리

1

클라이언트가 서버를 서브프로세스로 실행

uv run server.py 같은 명령으로 서버를 시작합니다.

↓
2

stdin으로 메시지 전송

클라이언트는 서버의 stdin에 JSON 메시지를 씁니다.

↓
3

stdout으로 응답 수신

서버는 stdout에 응답을 씁니다. 서버도 언제든지 먼저 메시지를 시작할 수 있습니다.

MCP 연결 핸드셰이크

모든 MCP 연결은 정해진 3단계 순서로 시작해야 합니다:

1

Initialize Request (클라이언트 → 서버)

클라이언트가 먼저 보내는 첫 번째 메시지

↓
2

Initialize Result (서버 → 클라이언트)

서버가 자신의 기능(capabilities)을 담아 응답

↓
3

Initialized Notification (클라이언트 → 서버)

클라이언트가 초기화 완료를 확인 (서버 응답 없음)

💡

핸드셰이크 후에만 도구 호출 가능

이 3단계 핸드셰이크 완료 후에야 도구 호출, 프롬프트 목록 요청 등 다른 작업을 수행할 수 있습니다.

4가지 통신 패턴

패턴방향채널
클라이언트 요청클라이언트 → 서버서버의 stdin에 씀
서버 응답서버 → 클라이언트서버의 stdout에 씀
서버 요청 (Sampling 등)서버 → 클라이언트서버의 stdout에 씀
클라이언트 응답클라이언트 → 서버서버의 stdin에 씀
⚠️

STDIO의 한계: 동일 머신 전용

STDIO 트랜스포트는 클라이언트와 서버가 반드시 같은 머신에서 실행되어야 합니다. 원격 배포가 필요하다면 StreamableHTTP 트랜스포트를 사용해야 합니다.

  • STDIO의 양방향 통신이 쉬운 이유는 무엇인가요? HTTP에서 이것이 어려운 이유와 비교해 보세요.
레슨 7 · StreamableHTTP

StreamableHTTP 트랜스포트: 원격 MCP 서버

HTTP를 통해 원격으로 호스팅된 MCP 서버에 연결하는 방법과 SSE를 이용한 양방향 통신 구현 원리를 이해합니다.

🎯 학습 목표

  • StreamableHTTP가 HTTP의 단방향 제약을 극복하는 방법을 설명할 수 있다
  • SSE(Server-Sent Events) 연결의 역할과 이중 SSE 연결 모델을 이해한다
  • stateless_http와 json_response 플래그의 영향을 파악한다

💡 핵심 요약

  • StreamableHTTP는 HTTP로 원격 MCP 서버에 접속 가능하게 하는 트랜스포트
  • 핵심 문제: HTTP는 클라이언트→서버 방향이 기본, 서버→클라이언트는 어려움
  • 해결책: SSE(Server-Sent Events)를 이용한 지속적 연결로 서버→클라이언트 통신 가능
  • 도구 호출 시 두 개의 SSE 연결 생성: Primary (서버 요청용) + Tool-Specific (도구 결과용)
  • stateless_http=True 또는 json_response=True는 SSE 우회 메커니즘을 깨뜨림

HTTP의 근본적 문제

방향HTTP 지원 여부이유
클라이언트 → 서버 요청✅ 쉬움서버는 알려진 URL을 가짐
서버 → 클라이언트 응답✅ 쉬움요청에 대한 응답
서버 → 클라이언트 요청❌ 어려움클라이언트는 알려진 URL이 없음
클라이언트 → 서버 응답 (역방향)❌ 어려움연결이 클라이언트 주도로만 시작 가능

이 HTTP 제약이 Sampling, Roots 요청, 진행률 알림 등 서버→클라이언트 통신이 필요한 MCP 기능에 영향을 줍니다.

StreamableHTTP의 SSE 우회 방식

1

초기화 + Session ID

클라이언트가 Initialize Request 전송 → 서버가 mcp-session-id 헤더를 포함한 결과 반환

↓
2

SSE 연결 수립

클라이언트가 GET 요청으로 SSE 연결 수립 — 서버가 언제든 이 채널로 메시지를 보낼 수 있음

↓
3

이중 SSE 연결 (도구 호출 시)

Primary SSE: 서버 요청 전용, 영구적 유지
Tool-Specific SSE: 도구 결과 전용, 결과 수신 후 종료

메시지 라우팅

메시지 유형사용되는 연결
진행률 알림Primary SSE 연결 (영구)
로그 메시지Tool-Specific SSE 연결 (임시)
도구 결과Tool-Specific SSE 연결 (임시)
Sampling 요청 (서버→클라이언트)Primary SSE 연결 (영구)
⚠️

설정 플래그가 기능을 망가뜨릴 수 있음

stateless_http=True 또는 json_response=True를 설정하면 SSE 기반 양방향 통신이 불가능해집니다. 진행률 알림, Sampling, 로그 등 서버→클라이언트 통신이 필요한 모든 기능이 중단됩니다.

레슨 8 · State Management

Stateless HTTP와 확장성 설계

수천 명의 클라이언트를 지원하는 프로덕션 MCP 서버를 구축할 때 상태 관리와 수평 확장의 트레이드오프를 이해합니다.

🎯 학습 목표

  • stateless_http와 json_response 플래그의 역할과 영향을 설명할 수 있다
  • 로드 밸런서 환경에서 상태 관리가 어려운 이유를 이해한다
  • 각 플래그를 사용해야 하는 시나리오를 파악한다

💡 핵심 요약

  • stateless_http=True: 수평 확장을 위해 세션 상태를 포기. Sampling, 진행률 알림 비활성화
  • json_response=True: SSE 스트리밍 없이 최종 결과만 JSON으로 반환
  • 로드 밸런서 환경에서 GET(SSE)과 POST(도구 호출)가 다른 서버 인스턴스로 라우팅될 수 있어 조정 문제 발생
  • Stateless HTTP의 유일한 장점: 클라이언트 초기화 핸드셰이크 불필요
  • 개발 시 프로덕션과 동일한 트랜스포트로 테스트해야 예기치 않은 문제를 조기에 발견 가능

확장성 문제

MCP 서버가 인기를 얻어 수천 명의 클라이언트를 처리해야 한다면:

1️⃣

단일 서버

소수 클라이언트에 적합. 상태 관리 쉬움

⚖️

로드 밸런서

수천 클라이언트를 여러 인스턴스에 분산

⚠️

조정 문제

GET(SSE)과 POST가 다른 인스턴스로 가면 Sampling 불가

두 플래그 비교

플래그효과포기하는 것얻는 것
stateless_http=True 세션 ID 없음, GET SSE 경로 비활성화 Sampling, 진행률 알림, 구독, 서버→클라이언트 요청 수평 확장, 로드 밸런서 호환, 초기화 핸드셰이크 불필요
json_response=True 스트리밍 없이 최종 JSON만 반환 중간 진행 메시지, 실행 중 로그, 스트리밍 응답 단순한 HTTP 응답, 비스트리밍 시스템 통합 용이

언제 어떤 플래그를 써야 할까?

Stateless HTTP를 사용할 때

  • 로드 밸런서를 사용한 수평 확장이 필요할 때
  • 서버→클라이언트 통신이 불필요한 경우
  • 도구가 AI 모델 Sampling을 요구하지 않을 때
  • 연결 오버헤드를 최소화하려 할 때

JSON Response를 사용할 때

  • 스트리밍 응답이 불필요한 경우
  • 표준 JSON을 기대하는 시스템과 통합할 때
  • 단순한 HTTP 요청-응답 패턴을 선호할 때
⚠️

개발 환경 ≠ 프로덕션 환경

로컬 STDIO로 개발하다가 HTTP로 배포하면 동작이 달라질 수 있습니다. Stateful과 Stateless 모드의 차이가 크므로, 개발 단계부터 프로덕션과 동일한 트랜스포트 설정으로 테스트해야 합니다.

📋 코스 전체 요약

  • Sampling: 서버가 클라이언트를 통해 AI 기능을 위임 — API 키·비용 없이
  • 알림: context.info()와 report_progress()로 실시간 UX 향상
  • Roots: 파일 시스템 접근 권한 — 사용자 친화성 + 보안 경계
  • JSON 메시지: MCP는 양방향 프로토콜 — 서버도 클라이언트에 요청 가능
  • STDIO: 개발/테스트에 최적. 동일 머신 전용
  • StreamableHTTP: 원격 배포 가능. SSE로 양방향 통신 구현
  • Stateless/JSON Response: 확장성을 위해 일부 기능을 포기하는 트레이드오프
