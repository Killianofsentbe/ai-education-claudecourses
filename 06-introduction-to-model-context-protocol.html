코스 개요

이 코스는 Model Context Protocol(MCP)의 기초부터 실습까지 체계적으로 배우는 입문 과정입니다. MCP가 무엇인지, 왜 필요한지, 그리고 어떻게 직접 구현하는지를 단계적으로 학습합니다.

실습 환경으로는 Python과 UV 패키지 매니저를 사용합니다. UV는 빠르고 현대적인 Python 패키지 관리 도구로, 이 코스에서 사용하는 모든 의존성을 간편하게 관리할 수 있습니다.

LESSON 02

MCP 소개

Introducing MCP
이 레슨을 마치면 할 수 있는 것
  • MCP가 해결하는 핵심 문제를 설명할 수 있다
  • MCP의 기본 아키텍처(클라이언트-서버 구조)를 이해할 수 있다
  • MCP와 일반 도구 사용(Tool Use)의 차이점을 구분할 수 있다
  • MCP 서버가 제공하는 세 가지 핵심 요소를 파악할 수 있다
핵심 요약
  • MCP는 커뮤니케이션 레이어로, Claude에게 도구와 컨텍스트를 제공하면서도 복잡한 통합 코드 작성을 피하게 해준다
  • MCP 서버는 도구(Tools), 프롬프트(Prompts), 리소스(Resources)를 표준화된 방식으로 노출한다
  • GitHub처럼 방대한 기능을 가진 서비스를 통합할 때, MCP는 이미 구현된 도구 정의를 재사용할 수 있게 한다
  • 서비스 제공자(AWS, GitHub 등)가 공식 MCP 서버를 직접 배포하는 경우도 많다
  • MCP와 도구 사용은 보완적이며, 누가 도구를 구현하는가가 핵심 차이점이다

MCP가 해결하는 문제

사용자가 "내 모든 저장소에 열려 있는 Pull Request가 뭐야?"라고 물었을 때, Claude는 GitHub API에 접근하는 도구가 필요합니다. GitHub는 저장소, PR, 이슈, 프로젝트 등 방대한 기능을 제공합니다. MCP 없이는 이 모든 기능에 대한 도구 스키마와 함수를 직접 작성, 테스트, 유지보수해야 합니다. 이는 엄청난 노력과 지속적인 유지보수 부담을 의미합니다.

Without MCP
직접 구현
모든 도구 스키마를 수동으로 작성, 유지보수 부담 높음
With MCP
재사용
기존 MCP 서버의 도구 정의를 그대로 활용, 통합 비용 절감

MCP 서버의 세 가지 핵심 요소

요소 설명 제어 주체
Tools Claude가 호출할 수 있는 함수들 (외부 API 호출, 계산 등) Claude (모델)
Resources 읽기 전용 데이터 접근 (파일, DB 조회 등) 애플리케이션
Prompts 미리 정의된 고품질 프롬프트 템플릿 사용자

자주 묻는 질문

흔한 오해
"MCP는 그냥 도구 사용(Tool Use)과 같은 거 아닌가요?" — MCP와 도구 사용은 보완적이지만 다른 개념입니다. 도구 사용은 Claude가 어떻게 도구를 호출하는지에 관한 것이고, MCP는 누가 도구를 정의하고 실행하는지에 관한 것입니다. MCP를 사용하면 다른 누군가가 이미 도구를 구현해 두었습니다.
생각해 볼 것들
  • 내가 자주 사용하는 외부 서비스(Slack, GitHub, Google Drive 등)에 이미 공식 MCP 서버가 있다면 어떻게 활용할 수 있을까?
  • MCP 서버를 직접 만들어야 할 상황은 언제이고, 기존 서버를 사용하면 충분한 상황은 언제인가?
  • 도구 정의를 외부 MCP 서버에 맡기는 것의 장단점은 무엇인가?
LESSON 03

MCP 클라이언트

MCP Clients
이 레슨을 마치면 할 수 있는 것
  • MCP 클라이언트의 역할과 책임을 설명할 수 있다
  • 전송 방식 독립성(Transport Agnostic)의 의미를 이해할 수 있다
  • ListTools 및 CallTool 메시지 타입의 용도를 구분할 수 있다
  • 사용자 질문이 시스템을 통해 처리되는 전체 흐름(12단계)을 설명할 수 있다
핵심 요약
  • MCP 클라이언트는 서버와 애플리케이션 사이의 통신 브리지 역할을 한다
  • MCP는 전송 방식 독립적으로, stdio, HTTP, WebSocket 등 다양한 프로토콜을 지원한다
  • 주요 메시지 타입은 ListToolsRequest/Result와 CallToolRequest/Result 두 쌍이다
  • 클라이언트는 프로토콜 세부 사항을 추상화하여 애플리케이션 로직에 집중할 수 있게 한다

전송 방식 독립성

MCP의 강점 중 하나는 전송 방식 독립적(Transport Agnostic)이라는 점입니다. 클라이언트와 서버는 설정에 따라 서로 다른 프로토콜로 통신할 수 있습니다.

  • 가장 일반적인 설정: 같은 머신에서 stdio(표준 입출력)를 통해 통신
  • 네트워크를 통한 연결: HTTP 또는 WebSocket 사용 가능

사용자 질문 처리 전체 흐름

"내 저장소 목록이 뭐야?"라는 질문이 시스템을 통해 처리되는 과정입니다:

01
사용자 질문 수신: 사용자가 서버에 질문을 제출
02
도구 탐색: 서버가 Claude에게 보낼 도구 목록 필요
03
ListTools 교환: 서버가 MCP 클라이언트에게 도구 목록 요청
04
MCP 통신: 클라이언트가 ListToolsRequest를 MCP 서버에 보내고 ListToolsResult를 받음
05
Claude 요청: 서버가 사용자 질문과 도구 목록을 Claude에게 전송
06
도구 사용 결정: Claude가 질문 답변을 위해 특정 도구 호출 결정
07
도구 실행 요청: 서버가 MCP 클라이언트에게 Claude가 지정한 도구 실행 요청
08
외부 API 호출: MCP 클라이언트가 CallToolRequest를 보내고 서버가 GitHub API 실제 호출
09
결과 반환: GitHub 데이터가 CallToolResult로 흐름을 따라 되돌아옴
10
도구 결과를 Claude에게: 서버가 도구 결과를 Claude에게 전송
11
최종 응답: Claude가 저장소 데이터를 활용해 최종 답변 작성
12
사용자에게 전달: 서버가 Claude의 응답을 사용자에게 전달
핵심 포인트
단계가 많아 보이지만, 각 컴포넌트는 명확한 책임을 가집니다. MCP 클라이언트는 서버 통신의 복잡성을 추상화하여 강력한 외부 도구에 접근하면서도 애플리케이션 로직에 집중할 수 있게 합니다.
생각해 볼 것들
  • 이 긴 통신 흐름에서 병목이 발생할 수 있는 지점은 어디인가?
  • 같은 머신에서 stdio로 통신하는 것과 HTTP로 원격 통신하는 것의 트레이드오프는?
  • MCP 클라이언트가 없다면 이 흐름을 어떻게 직접 구현해야 할까?

LESSON 05

MCP로 도구 정의하기

Defining Tools with MCP
이 레슨을 마치면 할 수 있는 것
  • Python MCP SDK를 사용하여 MCP 서버를 초기화할 수 있다
  • 데코레이터 패턴으로 MCP 도구를 정의하고 스키마를 자동 생성할 수 있다
  • Pydantic Field를 활용해 파라미터 설명을 추가할 수 있다
  • 문서 읽기 및 편집 도구를 완성할 수 있다
핵심 요약
  • FastMCP 클래스를 사용하면 단 한 줄로 MCP 서버를 초기화할 수 있다
  • @mcp.tool() 데코레이터를 사용하면 JSON 스키마를 수동으로 작성할 필요가 없다
  • Python 타입 힌트가 자동 검증을 제공하며 SDK가 적절한 스키마를 생성한다
  • Pydantic의 Field 클래스로 파라미터 설명을 추가하면 Claude가 도구 사용법을 정확히 이해한다
  • Python 예외(Exception)가 도구 오류 처리로 자연스럽게 통합된다

MCP 서버 초기화

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("DocumentMCP", log_level="ERROR")

# 인메모리 문서 저장소
docs = {
    "deposition.md": "This deposition covers the testimony of Angela Smith, P.E.",
    "report.pdf": "The report details the state of a 20m condenser tower.",
    "financials.docx": "These financials outline the project's budget and expenditures",
    "plan.md": "The plan outlines the steps for the project's implementation.",
    "spec.txt": "These specifications define the technical requirements..."
}

문서 읽기 도구

@mcp.tool(
    name="read_doc_contents",
    description="Read the contents of a document and return it as a string."
)
def read_document(
    doc_id: str = Field(description="Id of the document to read")
):
    if doc_id not in docs:
        raise ValueError(f"Doc with id {doc_id} not found")
    return docs[doc_id]

문서 편집 도구

@mcp.tool(
    name="edit_document",
    description="Edit a document by replacing a string in the document content."
)
def edit_document(
    doc_id: str = Field(description="Id of the document to edit"),
    old_str: str = Field(description="Text to replace. Must match exactly."),
    new_str: str = Field(description="New text to insert in place of the old text.")
):
    if doc_id not in docs:
        raise ValueError(f"Doc with id {doc_id} not found")
    docs[doc_id] = docs[doc_id].replace(old_str, new_str)

SDK 접근 방식의 이점

  • JSON 스키마를 수동으로 작성할 필요 없음
  • 타입 힌트가 자동 검증 제공
  • 명확한 파라미터 설명으로 Claude가 도구 사용법을 정확히 이해
  • Python 예외를 통한 자연스러운 오류 처리
  • 데코레이터를 통한 자동 도구 등록
생각해 볼 것들
  • 수동 JSON 스키마 작성 대비 SDK 데코레이터 방식의 유지보수 측면 차이는?
  • 파라미터 description이 Claude의 도구 선택에 얼마나 중요한 역할을 할까?
  • 이 문서 관리 서버 외에 어떤 종류의 MCP 서버를 만들어 볼 수 있을까?
LESSON 06

서버 인스펙터

The Server Inspector
이 레슨을 마치면 할 수 있는 것
  • MCP 인스펙터를 실행하고 브라우저에서 접속할 수 있다
  • 인스펙터를 통해 도구를 테스트하고 결과를 확인할 수 있다
  • 도구 상호작용과 상태 관리를 검증하는 개발 워크플로우를 구축할 수 있다
핵심 요약
  • Python MCP SDK에는 브라우저 기반 인스펙터가 내장되어 있어 별도 클라이언트 없이 테스트 가능하다
  • mcp dev mcp_server.py 명령으로 개발 서버와 인스펙터를 함께 실행한다
  • 인스펙터는 Resources, Tools, Prompts 탭으로 구성된다
  • 서버 상태가 도구 호출 사이에 유지되어 복잡한 워크플로우 테스트가 가능하다

인스펙터 시작하기

# 인스펙터 실행
mcp dev mcp_server.py

# 브라우저에서 접속: http://127.0.0.1:6274

인스펙터 사용 흐름

1

리소스 정의하기

Defining Resources
이 레슨을 마치면 할 수 있는 것
  • MCP 리소스의 개념과 도구와의 차이를 설명할 수 있다
  • 직접 리소스(Direct Resource)와 템플릿 리소스(Templated Resource)를 구분하고 구현할 수 있다
  • MIME 타입을 적절히 지정하여 클라이언트가 데이터를 올바르게 처리하도록 할 수 있다
핵심 요약
  • 리소스는 읽기 전용 데이터 접근을 위한 것으로, HTTP 서버의 GET 핸들러와 유사하다
  • 직접 리소스는 정적 URI를 사용하고, 템플릿 리소스는 URI에 파라미터를 포함한다
  • SDK가 자동으로 반환값을 직렬화하므로 JSON 변환을 수동으로 할 필요가 없다
  • 리소스는 @멘션 자동완성 UI나 프롬프트 컨텍스트 보강에 활용된다

직접 리소스 — 정적 URI

@mcp.resource(
    "docs://documents",
    mime_type="application/json"
)
def list_docs() -> list[str]:
    return list(docs.keys())

템플릿 리소스 — 동적 URI

@mcp.resource(
    "docs://documents/{doc_id}",
    mime_type="text/plain"
)
def fetch_doc(doc_id: str) -> str:
    if doc_id not in docs:
        raise ValueError(f"Doc with id {doc_id} not found")
    return docs[doc_id]

MIME 타입용도예시
application/json구조화된 데이터문서 목록, 설정
text/plain일반 텍스트문서 내용, 로그
application/pdf바이너리 파일PDF 문서

Accessing Resources
이 레슨을 마치면 할 수 있는 것
  • MCP 클라이언트에서 read_resource() 함수를 구현할 수 있다
  • MIME 타입에 따라 응답을 적절히 처리할 수 있다

핵심 요약
  • read_resource() 함수는 AnyUrl로 URI를 변환하여 MCP 서버에 요청한다
  • 응답은 MIME 타입에 따라 다르게 처리된다 (JSON 파싱 vs 텍스트 반환)
  • 리소스를 활용하면 @멘션으로 별도 도구 호출 없이 문서 내용을 즉시 주입할 수 있다
  • 리소스가 초기 컨텍스트에 포함되므로 AI 모델이 즉시 데이터에 응답 가능하다

리소스 읽기 함수 구현

import json
from pydantic import AnyUrl

async def read_resource(self, uri: str) -> Any:
    result = await self.session().read_resource(AnyUrl(uri))
    resource = result.contents[0]

    if isinstance(resource, types.TextResourceContents):
        if resource.mimeType == "application/json":
            return json.loads(resource.text)

    return resource.text

사용자가 /format doc_id를 입력하면 문서를 마크다운으로 변환하는 예제:

@mcp.prompt(
    name="format",
    description="Rewrites the contents of the document in Markdown format."
)
def format_document(
    doc_id: str = Field(description="Id of the document to format")
) -> list[base.Message]:
    prompt = f"""
Your goal is to reformat a document to be written with markdown syntax.

The id of the document you need to reformat is:
<document_id>
{doc_id}
</document_id>

Add in headers, bullet points, tables, etc as necessary. Feel free to add structure.
Use the 'edit_document' tool to edit the document.
"""

    return [base.UserMessage(prompt)]

Prompts in the Client

  • get_prompt()는 이름과 인수 딕셔너리를 받아 변수가 보간된 메시지 리스트를 반환한다
  • 인수 딕셔너리 (예: {"doc_id": "plan.md"})가 프롬프트 함수의 키워드 인수로 전달된다

클라이언트에서의 프롬프트 함수 구현

# 프롬프트 목록 조회
async def list_prompts(self) -> list[types.Prompt]:
    result = await self.session().list_prompts()
    return result.prompts

# 개별 프롬프트 가져오기 (변수 보간 포함)
async def get_prompt(self, prompt_name, args: dict[str, str]):
    result = await self.session().get_prompt(prompt_name, args)
    return result.messages

  • Tools(도구)는 모델이 제어한다 — Claude가 자율적으로 언제 호출할지 결정
  • Resources(리소스)는 앱이 제어한다 — 애플리케이션 코드가 언제 데이터를 가져오고 사용할지 결정
  • Prompts(프롬프트)는 사용자가 제어한다 — 사용자가 버튼 클릭, 슬래시 명령 등으로 트리거
  • 세 프리미티브는 각각 모델, 앱, 사용자 계층을 담당한다

세 가지 서버 프리미티브 비교

프리미티브 제어 주체 활용 시점 실제 예시
Tools Claude (모델) Claude에게 새로운 능력이 필요할 때 코드 실행, GitHub API 호출, 계산
Resources 애플리케이션 UI나 컨텍스트 보강을 위한 데이터 자동완성 목록, Google Drive 문서
Prompts 사용자 미리 정의된 워크플로우 제공 슬래시 명령, 워크플로우 버튼

프리미티브 선택 가이드

Use Tools When
Claude에게 새 능력 부여
Claude가 자율적으로 사용할 함수 필요. 예: 계산, API 호출, 파일 접근
Use Resources When
앱에서 데이터 가져오기
UI 표시나 컨텍스트 보강을 위한 데이터. 예: @멘션, 문서 목록
Use Prompts When
사용자 워크플로우 정의
사용자가 온디맨드 트리거. 예: /format, /summarize 명령

Claude 인터페이스에서의 실제 사례

  • 채팅 입력창 하단 워크플로우 버튼 → Prompts 예시
  • Google Drive 문서 추가 기능 → Resources 예시
  • Claude가 코드를 실행하거나 계산을 수행할 때 → Tools 예시
