LESSON 01

API 기초 & Claude 모델 이해

Anthropic API의 작동 원리와 Claude 모델 패밀리를 이해합니다

이 레슨에서 배우는 것

  • API 요청이 처리되는 5단계 흐름 이해
  • Claude 모델 패밀리 (Opus, Sonnet, Haiku) 비교
  • 토큰화 → 임베딩 → 문맥 반영 → 생성의 내부 처리 과정
  • 왜 API 키를 서버에서만 사용해야 하는지 보안 원칙

API 요청의 5단계 흐름

사용자가 채팅 인터페이스에서 "전송"을 클릭하는 순간부터 Claude의 응답이 화면에 나타나기까지, 모든 상호작용은 예측 가능한 5단계를 거칩니다.

1
클라이언트 → 서버

사용자의 브라우저나 앱이 여러분의 서버로 요청을 보냅니다. API 키는 절대 클라이언트에 노출되어서는 안 됩니다.

2
서버 → Anthropic API

여러분의 서버가 보안 저장된 API 키를 사용해 Anthropic API에 요청을 전달합니다.

3
Claude 내부 처리

토큰화 → 임베딩 → 문맥 반영 → 생성의 4단계로 응답을 생성합니다.

4
API → 서버 응답

생성된 텍스트, 토큰 사용량, 중단 이유가 포함된 구조화된 응답을 반환합니다.

5
서버 → 클라이언트

서버가 생성된 텍스트를 클라이언트 앱으로 전달하여 사용자에게 표시합니다.

Claude 내부 처리 4단계

Anthropic API에 요청이 도착하면 Claude는 다음 4단계로 응답을 생성합니다:

1. 토큰화 (Tokenization)

입력 텍스트를 토큰이라는 작은 단위로 분해합니다. 토큰은 단어, 단어의 일부, 공백, 기호일 수 있습니다. 간단히 말해 각 단어를 하나의 토큰으로 이해하면 됩니다.

2. 임베딩 (Embedding)

각 토큰을 숫자 목록(임베딩)으로 변환합니다. 이 숫자들이 단어의 의미를 포착합니다. 예: "quantum"은 물리학, 양자역학, 극소량 등 여러 의미를 가질 수 있습니다.

3. 문맥 반영 (Contextualization)

주변 단어를 기반으로 각 임베딩을 정제합니다. 문맥에 맞는 의미가 강조되도록 수치 표현을 조정합니다.

4. 생성 (Generation)

문맥이 반영된 임베딩이 출력 레이어를 통과해 다음 단어의 확률을 계산합니다. Claude는 확률과 제어된 무작위성을 조합해 자연스러운 응답을 생성합니다.

Claude 모델 패밀리 비교

모델특성적합한 용도비용/속도
Claude Opus최고 성능, 가장 강력한 추론 능력복잡한 분석, 연구, 전략적 작업높음 / 느림
Claude Sonnet성능과 속도의 균형대부분의 프로덕션 사용 사례중간 / 빠름
Claude Haiku가장 빠르고 경제적데이터 생성, 간단한 분류, 높은 처리량낮음 / 매우 빠름
⚠️ 보안 원칙

API 키를 클라이언트 코드에 직접 포함하면 심각한 보안 취약점이 생깁니다. 누구든 키를 추출해 무단으로 요청을 보낼 수 있습니다. 항상 .env 파일에 저장하고 .gitignore에 추가하세요.

필수 API 파라미터

모든 요청에는 다음 필드가 포함되어야 합니다:

  • api_key — Anthropic에 요청을 식별하는 인증 키
  • model — 사용할 모델 이름 (예: "claude-sonnet-4-0")
  • messages — 사용자 입력을 담는 메시지 리스트
  • max_tokens — Claude가 생성할 수 있는 최대 토큰 수 (안전 제한)

핵심 요약

  • API 요청은 항상 클라이언트 → 내 서버 → Anthropic API 순서로 중개됩니다 (키 보안을 위해)
  • Claude는 토큰화 → 임베딩 → 문맥 반영 → 생성의 4단계로 응답을 만듭니다
  • Opus(복잡성) / Sonnet(균형) / Haiku(속도·비용) 용도에 맞게 모델을 선택하세요
  • max_tokens는 목표치가 아니라 상한선입니다. Claude는 그 안에서 적절히 멈춥니다
  • 생성 중단 이유: max_tokens 도달, 자연스러운 종료, 중단 시퀀스 감지

성찰 질문

  • 내 사용 사례에서 Claude Sonnet과 Haiku 중 어느 것이 더 적합할까? 기준은 무엇인가?
  • API 키를 환경 변수로 관리하는 것이 왜 단순한 관행을 넘어 필수 원칙인가?
  • 토큰 비용을 최소화하면서도 충분한 응답 품질을 얻으려면 max_tokens를 어떻게 설정해야 할까?
LESSON 02

첫 API 요청 & 대화 관리

client.messages.create()로 첫 요청을 만들고, 멀티턴 대화를 올바르게 관리합니다

이 레슨에서 배우는 것

  • Python SDK로 첫 API 요청 구성 및 실행
  • Claude가 대화 히스토리를 저장하지 않는다는 핵심 개념 이해
  • 멀티턴 대화를 위한 메시지 히스토리 관리 패턴
  • 응답 객체 구조와 텍스트 추출 방법

환경 설정 & 첫 요청

# 패키지 설치
%pip install anthropic python-dotenv

# .env 파일에 API 키 저장 (절대 코드에 직접 쓰지 마세요)
# ANTHROPIC_API_KEY="sk-ant-..."

from dotenv import load_dotenv
from anthropic import Anthropic

load_dotenv()
client = Anthropic()
model = "claude-sonnet-4-0"

# 첫 번째 API 요청
message = client.messages.create(
    model=model,
    max_tokens=1000,
    messages=[
        {
            "role": "user",
            "content": "What is quantum computing? Answer in one sentence"
        }
    ]
)

# 응답 텍스트 추출
print(message.content[0].text)

Claude의 상태 없음(Stateless) 특성

⚠️ 핵심 개념: Claude는 대화 히스토리를 저장하지 않습니다!

각 요청은 완전히 독립적입니다. "다른 문장을 써줘"라고 물으면 Claude는 무엇에 대한 문장인지 전혀 모릅니다. 이전 대화 내용을 매번 함께 전송해야 합니다.

멀티턴 대화 관리 패턴

# 대화 관리를 위한 헬퍼 함수들
def add_user_message(messages, text):
    messages.append({"role": "user", "content": text})

def add_assistant_message(messages, text):
    messages.append({"role": "assistant", "content": text})

def chat(messages):
    message = client.messages.create(
        model=model,
        max_tokens=1000,
        messages=messages,
    )
    return message.content[0].text

# 멀티턴 대화 예시
messages = []

# 1. 첫 번째 질문
add_user_message(messages, "Define quantum computing in one sentence")
answer = chat(messages)

# 2. 응답을 히스토리에 추가 (중요!)
add_assistant_message(messages, answer)

# 3. 후속 질문 (Claude가 이전 맥락을 알 수 있음)
add_user_message(messages, "Write another sentence expanding on that")
final_answer = chat(messages)
💡 핵심 패턴

Claude의 응답을 받은 후에는 반드시 add_assistant_message()로 히스토리에 추가하세요. 그래야 다음 요청 시 Claude가 이전 맥락을 인식할 수 있습니다.

응답 객체 구조

필드설명접근 방법
content[0].text생성된 텍스트 (가장 자주 사용)message.content[0].text
usage.input_tokens입력 토큰 수비용 추적에 활용
usage.output_tokens출력 토큰 수비용 추적에 활용
stop_reason생성 중단 이유end_turn, max_tokens, stop_sequence
model사용된 모델 이름응답 검증에 활용

핵심 요약

  • Claude는 무상태(stateless) — 모든 대화 히스토리를 매번 요청에 포함해야 합니다
  • add_user_message() / add_assistant_message() 헬퍼 함수로 히스토리를 체계적으로 관리하세요
  • 응답 텍스트는 message.content[0].text로 간단히 추출합니다
  • 토큰 사용량을 추적해 비용을 모니터링하는 습관을 들이세요
  • stop_reason을 확인해 응답이 완전히 끝났는지 확인하세요

성찰 질문

  • 대화 히스토리가 길어질수록 토큰 비용이 증가합니다. 히스토리를 어떻게 효율적으로 관리할 수 있을까?
  • 대화 히스토리를 데이터베이스에 저장하고 불러오는 시스템을 어떻게 설계할까?
LESSON 03

시스템 프롬프트 & 출력 제어

시스템 프롬프트로 Claude의 역할을 정의하고, Temperature·스트리밍·구조화 출력으로 응답을 제어합니다

이 레슨에서 배우는 것

  • 시스템 프롬프트로 Claude의 페르소나와 행동 방식 정의
  • Temperature 파라미터로 창의성과 일관성 사이 균형 조절
  • 스트리밍으로 사용자 경험 개선 (실시간 텍스트 표시)
  • 어시스턴트 메시지 프리필 + 중단 시퀀스로 구조화 출력 생성

시스템 프롬프트 활용

시스템 프롬프트는 Claude가 어떻게 응답해야 하는지 안내합니다. 역할, 스타일, 제한 사항을 정의하는 데 사용합니다.

system_prompt = """
You are a patient math tutor.
Do not directly answer a student's questions.
Guide them to a solution step by step.
Show examples of similar problems when helpful.
"""

def chat(messages, system=None):
    params = {
        "model": model,
        "max_tokens": 1000,
        "messages": messages,
    }
    if system:
        params["system"] = system

    message = client.messages.create(**params)
    return message.content[0].text

# 시스템 프롬프트와 함께 호출
answer = chat(messages, system=system_prompt)
💡 중요

Claude의 API는 system=None을 받지 않습니다. 조건부로 파라미터를 추가하도록 if system:으로 처리하세요.

Temperature 파라미터

Temperature는 0~1 사이의 값으로 Claude 응답의 예측 가능성과 창의성을 조절합니다.

0.0 – 0.3
낮은 온도 (결정적)
  • 팩트 응답
  • 코딩 지원
  • 데이터 추출
  • 콘텐츠 검토
0.4 – 0.7
중간 온도 (균형)
  • 요약
  • 교육 콘텐츠
  • 문제 해결
  • 제약 있는 창작
0.8 – 1.0
높은 온도 (창의적)
  • 브레인스토밍
  • 창의적 글쓰기
  • 마케팅 문안
  • 아이디어 다양화
def chat(messages, system=None, temperature=1.0):
    params = {
        "model": model,
        "max_tokens": 1000,
        "messages": messages,
        "temperature": temperature
    }
    if system:
        params["system"] = system
    message = client.messages.create(**params)
    return message.content[0].text

# 낮은 온도 - 일관된 결과
answer = chat(messages, temperature=0.0)

# 높은 온도 - 다양한 창의적 결과
answer = chat(messages, temperature=1.0)

스트리밍으로 UX 개선

응답이 10-30초 걸릴 수 있습니다. 스트리밍은 텍스트가 생성되는 대로 실시간으로 보여줍니다.

# 스트리밍 구현
with client.messages.stream(
    model=model,
    max_tokens=1000,
    messages=messages
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)  # 실시간 출력

    # 완전한 메시지 객체 (DB 저장용)
    final_message = stream.get_final_message()

구조화 출력: 프리필 + 중단 시퀀스

어시스턴트 메시지 프리필과 중단 시퀀스를 조합하면 JSON 등 순수 구조화 데이터를 얻을 수 있습니다.

messages = []
add_user_message(messages, "Generate an AWS EventBridge rule as JSON")
add_assistant_message(messages, "```json")  # JSON 시작을 프리필

# 닫는 코드 블록이 나오면 즉시 중단
text = chat(messages, stop_sequences=["```"])

# 결과: 순수 JSON만 반환 (마크다운 없음)
import json
data = json.loads(text)
💡 활용 패턴

이 기법은 웹 앱에서 사용자가 바로 복사할 수 있는 순수 코드/JSON이 필요할 때 특히 유용합니다. Claude의 설명 텍스트를 제거하고 원하는 형식만 정확히 얻을 수 있습니다.

핵심 요약

  • 시스템 프롬프트는 Claude의 역할·스타일·제한을 정의하는 강력한 도구입니다
  • Temperature: 낮음(0~0.3)은 일관성, 높음(0.8~1)은 창의성에 적합합니다
  • 스트리밍은 사용자 경험을 크게 개선하며 client.messages.stream()으로 구현합니다
  • 프리필 + 중단 시퀀스 조합은 순수 구조화 출력(JSON, 코드)을 보장합니다
  • 프리필된 어시스턴트 메시지는 응답의 시작점을 강제로 지정합니다

성찰 질문

  • 시스템 프롬프트와 사용자 메시지 중 어디에 제약 사항을 넣는 것이 더 효과적일까?
  • 스트리밍을 사용할 때 부분적으로 도착한 응답을 어떻게 처리해야 할까?
  • 다양한 사용 사례에서 최적의 Temperature를 어떻게 실험적으로 결정할 수 있을까?
LESSON 04

프롬프트 평가 (Evals)

데이터 기반으로 프롬프트 성능을 측정하고 체계적으로 개선하는 Eval 워크플로우를 구축합니다

이 레슨에서 배우는 것

  • 왜 주관적 판단 대신 객관적 평가가 필요한지 이해
  • 5단계 Eval 워크플로우: 프롬프트 → 데이터셋 → 실행 → 채점 → 반복
  • Claude로 테스트 데이터셋 자동 생성
  • 모델 기반 채점과 코드 기반 채점의 차이와 활용

왜 Eval이 필요한가?

프롬프트를 변경할 때 "이게 더 나은가?"라는 질문에 직관으로 답하는 것은 위험합니다. Eval은 이 질문에 숫자로 답합니다.

Option 1: 수동 검토

몇 가지 예시를 직접 테스트합니다. 빠르지만 주관적이고 샘플이 적어 신뢰하기 어렵습니다.

Option 2: 히스토릭 테스트

이전 버전의 결과물과 비교합니다. 개선은 보이지만 절대적 품질을 판단하기 어렵습니다.

Option 3: Eval 파이프라인 ✓

프롬프트를 체계적인 평가 파이프라인에 실행합니다. 객관적 점수로 변경 전후를 정확히 비교합니다.

실제 점수 비교 예시

프롬프트 개선 전: 2.32점 → 명확성 추가 후: 3.92점 → 구체성 추가 후: 7.86점 (10점 만점)

5단계 Eval 워크플로우

1
프롬프트 초안 작성

테스트할 기준 프롬프트를 작성합니다. 처음에는 단순하게 시작해도 됩니다.

2
Eval 데이터셋 생성

실제 사용 사례를 대표하는 입력 샘플을 수동으로 만들거나 Claude로 자동 생성합니다.

3
Claude에 통과

데이터셋의 각 입력을 프롬프트와 결합해 Claude에게 보내고 출력을 수집합니다.

4
채점기로 평가

모델 기반 또는 코드 기반 채점으로 각 출력에 점수를 매깁니다 (1-10 척도).

5
프롬프트 수정 & 반복

명확성, 구체성, XML 태그, 예시 제공의 4가지 핵심 기법으로 프롬프트 품질을 극적으로 향상시킵니다

Tool Use (도구 사용)

Claude가 외부 함수와 API를 호출할 수 있도록 도구를 정의하고, 멀티턴 도구 대화와 구조화 데이터 추출을 구현합니다

from datetime import datetime

def get_current_datetime(date_format="%Y-%m-%d %H:%M:%S"):
    if not date_format:
        raise ValueError("date_format cannot be empty")
    return datetime.now().strftime(date_format)

def add_duration_to_datetime(dt_string, days=0, hours=0, minutes=0):
    from datetime import timedelta
    dt = datetime.strptime(dt_string, "%Y-%m-%d %H:%M:%S")
    result = dt + timedelta(days=days, hours=hours, minutes=minutes)
    return result.strftime("%Y-%m-%d %H:%M:%S")

def set_reminder(message, datetime_string):
    # 실제 구현에서는 알림 시스템과 연결
    return f"Reminder set: '{message}' for {datetime_string}"

LESSON 07

RAG (검색 증강 생성)

텍스트 청킹, 임베딩, 벡터 검색을 활용해 외부 문서를 Claude에게 효율적으로 제공하는 RAG 파이프라인을 구축합니다

이 레슨에서 배우는 것

  • RAG가 필요한 이유 — Claude의 컨텍스트 창 한계 극복
  • 3가지 청킹 전략: 구조 기반, 문장 기반, 크기 기반
  • 텍스트 임베딩과 코사인 유사도로 의미 기반 검색 구현
  • BM25 어휘 검색, 재랭킹, 컨텍스트 검색으로 정확도 향상

RAG가 필요한 이유

Claude는 학습 데이터 기준의 지식만 갖고 있습니다. 최신 정보, 사내 문서, 개인 데이터를 활용하려면 RAG가 필요합니다. 또한 모든 문서를 컨텍스트에 넣는 것은 비용과 성능 측면에서 비효율적입니다.

A
전처리 단계 (사전 실행)

문서 청킹 → 임베딩 생성 → 벡터 DB 저장. 이 작업은 한 번만 실행합니다.

B
쿼리 단계 (실시간 처리)

사용자 질문 임베딩 → 유사 청크 검색 → Claude에 관련 청크 + 질문 전달.

3가지 청킹 전략

전략방법장점적합한 경우
구조 기반 제목, 섹션, 헤더 기준으로 분할 의미론적으로 완전한 단위 유지 구조가 잘 잡힌 내부 문서, 기술 매뉴얼
문장 기반 문장 단위로 분할, 일정 개수씩 묶음 문법적으로 완전한 청크 일반 텍스트 문서, 블로그, 기사
크기 기반 (권장) 고정 문자 수로 분할, 오버랩 적용 모든 콘텐츠 유형에 적용 가능 코드, 다양한 형식의 문서, 프로덕션 환경 
def chunk_by_size(text, chunk_size=1000, overlap=200):
    chunks = []
    start = 0
    while start < len(text):
        end = min(start + chunk_size, len(text))
        chunks.append(text[start:end])
        start += chunk_size - overlap  # 오버랩으로 경계 문맥 유지
    return chunks

텍스트 임베딩 & 코사인 유사도

import voyageai

client_voyage = voyageai.Client()  # VOYAGE_API_KEY 필요

def generate_embedding(text, input_type="document"):
    result = client_voyage.embed([text], model="voyage-3-large", input_type=input_type)
    return result.embeddings[0]

# 코사인 유사도로 관련 청크 검색
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def find_relevant_chunks(query, chunk_embeddings, chunks, top_k=3):
    query_embedding = generate_embedding(query, input_type="query")
    similarities = [cosine_similarity(query_embedding, ce) for ce in chunk_embeddings]
    top_indices = np.argsort(similarities)[-top_k:][::-1]
    return [chunks[i] for i in top_indices]

고급 검색 전략

BM25 어휘 검색

키워드 매칭 기반 검색. "quantum" 같은 특수 용어나 정확한 단어 검색에 강합니다. 시맨틱 검색의 단점을 보완합니다.

하이브리드 검색

임베딩(시맨틱) + BM25(어휘) 결과를 결합합니다. 두 방식의 장점을 모두 활용해 검색 정확도를 높입니다.

재랭킹 (Reranking)

초기 검색 결과를 더 정밀한 모델로 재정렬합니다. VoyageAI의 rerank API 활용. 비용이 있지만 정확도가 크게 향상됩니다.

컨텍스트 검색

각 청크에 문서 맥락 요약을 추가합니다. Claude로 각 청크에 컨텍스트 정보를 자동 생성해 검색 정확도를 향상시킵니다.

핵심 요약

  • RAG = 문서 청킹 → 임베딩 저장 → 쿼리 시 관련 청크 검색 → Claude에 컨텍스트 제공
  • 프로덕션에서는 크기 기반 청킹(오버랩 포함)이 가장 안정적입니다
  • Anthropic은 자체 임베딩 서비스가 없으므로 VoyageAI를 권장합니다
  • 코사인 유사도로 시맨틱 검색, BM25로 어휘 검색 — 하이브리드가 최선입니다
  • 재랭킹은 비용이 있지만 검색 정확도를 크게 향상시킵니다

성찰 질문

LESSON 08

Claude 고급 기능

확장 사고, 이미지·PDF·인용, 프롬프트 캐싱 등 Claude의 강력한 고급 기능들을 실무에 적용합니다

이 레슨에서 배우는 것

  • Extended Thinking으로 복잡한 추론 문제 해결
  • 이미지·PDF 입력 처리와 효과적인 프롬프팅
  • Citations로 투명한 출처 추적 구현
  • 프롬프트 캐싱으로 비용 최대 90% 절감

Extended Thinking (확장 사고)

복잡한 수학, 다단계 추론, 전략적 분석에서 Claude가 응답 전 내부적으로 더 깊이 사고할 수 있도록 합니다.

def chat_with_thinking(messages, budget_tokens=10000):
    response = client.messages.create(
        model="claude-sonnet-4-0",
        max_tokens=16000,
        temperature=1,  # 확장 사고는 temperature=1 필수
        thinking={
            "type": "enabled",
            "budget_tokens": budget_tokens  # 사고에 할당할 최대 토큰
        },
        messages=messages
    )

    thinking_text = ""
    response_text = ""

    for block in response.content:
        if block.type == "thinking":
            thinking_text = block.thinking  # 내부 사고 과정
        elif block.type == "text":
            response_text = block.text      # 최종 응답

    return thinking_text, response_text
⚠️ 비용 고려

확장 사고는 더 많은 토큰을 사용합니다. 먼저 일반 프롬프트를 최적화하고, 그래도 부족할 때만 확장 사고를 활성화하세요.

💡 이미지 프롬프팅 팁

단순한 질문보다 단계별 분석 방법론을 제공하세요. "How many marbles?" 보다 "Step 1: Identify each marble one at a time. Step 2: Verify by counting from bottom-left..." 방식이 훨씬 정확합니다.

프롬프트 캐싱

반복적으로 사용하는 대용량 컨텍스트(시스템 프롬프트, RAG 문서 등)를 캐시해 비용과 지연을 줄입니다.

response = client.messages.create(
    model=model,
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": large_system_prompt,  # 수천 토큰의 시스템 프롬프트
            "cache_control": {"type": "ephemeral"}  # 캐시 마킹
        }
    ],
    messages=messages
)

# 사용량 확인
print(f"캐시 생성: {response.usage.cache_creation_input_tokens}")
print(f"캐시 히트: {response.usage.cache_read_input_tokens}")
💡 캐싱 효과

캐시 읽기는 일반 입력 토큰 비용의 약 10%만 청구됩니다. RAG 파이프라인에서 수천 토큰의 문서를 반복 전송할 때 비용을 최대 90% 절감할 수 있습니다.

핵심 요약

  • Extended Thinking은 복잡한 추론에 강력하지만 비용이 높습니다 — 최후 수단으로 활용하세요
  • 이미지/PDF는 base64 인코딩으로 전송하며, 단계별 분석 프롬프트가 정확도를 높입니다
  • Citations는 문서 기반 Q&A에서 투명성과 신뢰성을 크게 향상시킵니다
  • 프롬프트 캐싱은 반복 요청이 많은 프로덕션 환경에서 비용을 90%까지 절감합니다
  • 캐싱 활성화: cache_control: {"type": "ephemeral"}를 시스템 프롬프트나 문서 블록에 추가하세요

  • Citations가 사용자 신뢰에 어떤 영향을 미칠 수 있으며, 어떤 사용 사례에 가장 적합한가?

LESSON 09

MCP (모델 컨텍스트 프로토콜)

MCP의 아키텍처를 이해하고, 클라이언트·서버를 직접 구현해 외부 서비스와 통합하는 방법을 배웁니다

이 레슨에서 배우는 것

  • MCP가 일반 Tool Use와 다른 점 이해
  • MCP 클라이언트-서버 아키텍처와 메시지 타입
  • MCP 서버에서 도구, 리소스, 프롬프트 정의 방법
  • CLI 챗봇 프로젝트로 MCP 실습

MCP vs 일반 Tool Use

일반 Tool Use

개발자가 직접 도구 함수와 JSON Schema를 모두 작성해야 합니다. GitHub API를 연결하려면 수십 개의 함수를 직접 구현해야 합니다.

MCP

도구 정의와 실행이 MCP 서버에 패키징되어 있습니다. 누군가 이미 만들어둔 서버에 연결만 하면 됩니다. 개발자 생산성이 크게 향상됩니다.

MCP는 도구 사용의 대안이 아닌 보완적 개념입니다. "누가 도구를 만드느냐"의 문제를 해결합니다.

MCP 통신 흐름

1
ListToolsRequest → ListToolsResult

클라이언트가 서버에 "어떤 도구가 있나요?"를 물어봅니다.

2
도구 목록 + 사용자 쿼리 → Claude 전송

수집된 도구 스키마와 사용자 질문을 함께 Claude에게 전달합니다.

3
CallToolRequest → CallToolResult

Claude가 도구 호출을 요청하면, 클라이언트가 서버에 실제 실행을 요청합니다.

4
도구 결과 → Claude → 최종 응답

실행 결과를 Claude에게 전달하면 최종 사용자 응답을 생성합니다.

MCP 서버의 3가지 제공 요소

요소역할예시
Tools (도구) Claude가 호출할 수 있는 함수들. 데이터 조회, 수정, 외부 서비스 호출 등 read_document(doc_id), update_document(doc_id, content)
Resources (리소스) Claude가 읽을 수 있는 데이터 소스. 파일, DB 레코드, 설정 등 resource://docs/list, resource://docs/{id}
Prompts (프롬프트) 재사용 가능한 고품질 프롬프트 템플릿. 인수를 받아 동적 생성 format_document(doc_id), summarize_with_context(topic)

MCP 클라이언트 구현 (핵심 부분)

from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

class MCPClient:
    async def connect(self, server_command: list[str]):
        server_params = StdioServerParameters(command=server_command[0], args=server_command[1:])
        self._stdio, self._write = await stdio_client(server_params).__aenter__()
        self._session = ClientSession(self._stdio, self._write)
        await self._session.initialize()

    async def list_tools(self):
        result = await self._session.list_tools()
        return result.tools

    async def call_tool(self, name: str, arguments: dict):
        result = await self._session.call_tool(name, arguments)
        return result.content

    async def list_resources(self):
        result = await self._session.list_resources()
        return result.resources

    async def get_prompt(self, name: str, args: dict):
        result = await self._session.get_prompt(name, args)
        return result.messages

핵심 요약

  • MCP는 도구 구현의 부담을 서버로 이전합니다 — 연결만 하면 수백 개의 도구를 즉시 사용 가능
  • MCP 서버는 Tools·Resources·Prompts의 3가지 요소를 제공합니다
  • ListTools/CallTool 두 가지 핵심 메시지 타입으로 통신합니다
  • MCP는 STDIO, HTTP, WebSocket 등 다양한 트랜스포트를 지원합니다
  • 실제 프로젝트에서는 클라이언트 또는 서버 중 하나만 구현하는 경우가 대부분입니다

  • MCP 서버를 직접 만든다면 어떤 도구·리소스·프롬프트를 제공해야 가장 유용할까?
  • MCP 서버를 오픈소스로 공개하면 어떤 가치를 가질 수 있는가?
LESSON 10

에이전트 & Claude Code

자율적으로 작업을 수행하는 AI 에이전트의 핵심 특성을 이해하고, Claude Code와 Computer Use를 실무에 적용합니다

이 레슨에서 배우는 것

  • AI 에이전트의 정의와 핵심 특성 4가지
  • Claude Code의 작동 방식과 실무 활용
  • Computer Use로 데스크탑 환경 자동화
  • 에이전트 설계 시 고려해야 할 신뢰성과 안전성

에이전트란 무엇인가?

에이전트는 도구를 사용하고, 환경과 상호작용하며, 자율적으로 다단계 작업을 수행하는 AI 시스템입니다. Claude Code와 Computer Use가 대표적인 사례입니다.

1. 도구 통합

외부 서비스와 상호작용하기 위해 다양한 도구를 활용합니다. 파일 시스템, 웹 브라우저, API 등을 조합합니다.

2. 다단계 실행

단일 프롬프트-응답을 넘어 여러 단계의 작업을 자율적으로 계획하고 실행합니다.

3. 환경 상호작용

텍스트 대화를 넘어 파일, 데스크탑, 웹 등 실제 환경과 직접 상호작용합니다.

4. 자율적 문제 해결

명확한 지시 없이도 목표를 향해 스스로 접근 방식을 결정하고 반복적으로 개선합니다.

Claude Code

터미널 기반 코딩 어시스턴트로, 실제 개발 환경에서 파일 수정, 버그 수정, 코드 설명 등을 자율적으로 수행합니다.

기능설명
파일 편집코드베이스를 읽고 직접 수정하며 변경 사항을 설명합니다
버그 수정오류 메시지를 분석하고 근본 원인을 찾아 수정합니다
코드 질문 응답"이 함수는 어떻게 작동하나?"와 같은 질문에 컨텍스트 기반으로 답합니다
개발 워크플로우테스트 실행, Git 작업, 리팩토링 등 다양한 작업을 지원합니다
MCP 통합MCP 서버를 연결해 GitHub, Jira 등 외부 서비스와 연동합니다
💡 Claude Code 설정

npm install -g @anthropic-ai/claude-code로 설치 후, 터미널에서 claude를 입력하면 시작됩니다. ANTHROPIC_API_KEY가 환경 변수에 설정되어야 합니다.

Computer Use

스크린샷을 보고 화면을 이해하며, 키보드/마우스 조작으로 실제 데스크탑 환경을 자동화합니다.

    {"type": "bash_20241022", "name": "bash"}

response = client.messages.create(

⚠️ 보안 주의

Computer Use는 실제 시스템을 제어합니다. 격리된 환경(VM, 컨테이너)에서 실행하고, 민감한 계정이나 파일에 대한 접근을 제한하세요.

최소 권한 원칙

인간 감독 포함

오류 처리

실패 케이스를 명시적으로 처리하고, 루프에 빠지지 않도록 최대 시도 횟수를 제한하세요.

관찰 가능성
