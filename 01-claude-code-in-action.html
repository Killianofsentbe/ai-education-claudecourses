LESSON 01

코딩 어시스턴트의 원리 — Tool Use란?

⏱ 예상 소요 시간: 20분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 코딩 어시스턴트가 어떻게 파일을 읽고 코드를 수정하는지 원리를 설명할 수 있다
  • Tool Use(도구 사용) 메커니즘이 언어 모델의 한계를 어떻게 극복하는지 이해할 수 있다
  • Claude의 Tool Use 강점이 다른 모델과 차별화되는 이유를 설명할 수 있다
  • 보안·확장성·지속성 측면에서 Claude Code의 이점을 나열할 수 있다
핵심 요약
  • 언어 모델 자체는 파일을 읽거나 명령을 실행할 수 없습니다. 텍스트 입력→텍스트 출력이 전부입니다. 코딩 어시스턴트는 "도구 사용(Tool Use)" 시스템으로 이 한계를 극복합니다.
  • Tool Use 흐름: 사용자 요청 → 모델이 도구 호출 텍스트 반환 → 어시스턴트가 실제 도구 실행 → 결과를 모델에 전달 → 최종 답변 생성.
  • Claude는 Tool Use에 특히 강합니다. 처음 보는 도구도 빠르게 파악하고 복잡한 작업에 여러 도구를 조합합니다.
  • 보안 강점: 코드베이스 전체를 인덱싱하지 않아도 탐색 가능 — 코드 유출 리스크 감소.
  • 확장성: 새 도구를 추가하면 Claude가 자동으로 적응해 사용합니다.

코딩 어시스턴트는 어떻게 동작하는가?

에러 메시지 기반 버그 수정 작업을 AI에게 맡기면, 인간 개발자와 유사한 과정을 거칩니다:

단계 1
컨텍스트 수집
에러의 의미와 관련 파일 파악
단계 2
계획 수립
코드 수정 방법과 검증 방법 결정
단계 3
실행
파일 수정 및 명령 실행으로 솔루션 구현
핵심 문제
외부 환경 상호작용
1단계·3단계는 파일 읽기, 명령 실행 등 외부 접근 필요

Tool Use: 한계를 넘는 방법

# 사용자 요청:
"main.go 파일에 어떤 코드가 있어?"

# 어시스턴트가 모델에게 전달 (내부적):
"파일을 읽고 싶다면 'ReadFile: 파일명' 형식으로 응답하세요"

# 모델 응답:
ReadFile: main.go

# 어시스턴트가 실제 파일을 읽어 내용을 모델에 전달
# 모델이 파일 내용 기반으로 최종 답변 생성

Claude의 Tool Use가 특별한 이유

강점설명실제 이점
복잡한 작업여러 도구 조합으로 멀티스텝 작업 처리단순 검색을 넘어 실제 워크플로우 자동화
새 도구 적응처음 보는 도구도 설명만 있으면 즉시 활용MCP 서버 추가 시 별도 훈련 불필요
확장 가능성새 도구 추가 시 자동으로 통합개발 환경 진화에 자연스럽게 확장
보안필요한 파일만 접근코드 유출 리스크 최소화
생각해 볼 것들
  • 내 개발 워크플로우에서 Tool Use로 자동화할 수 있는 반복 작업은?
  • 코드베이스 전체를 외부 서버에 전송하지 않는 것이 내 프로젝트에서 왜 중요한가?
  • Claude가 처음 보는 도구를 스스로 파악한다는 것이 팀 확장에 어떤 의미인가?
LESSON 02

설치 및 프로젝트 설정

⏱ 예상 소요 시간: 20분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 운영체제별 Claude Code 설치 방법을 실행할 수 있다
  • AWS Bedrock 또는 Google Vertex AI 추가 설정을 할 수 있다
  • 실습 프로젝트를 로컬에 설정하고 Claude Code와 함께 사용할 수 있다
핵심 요약
  • 세 가지 설치 방법: macOS Homebrew, 범용 curl 스크립트, Windows CMD. 설치 후 터미널에서 claude를 실행하면 최초 인증이 시작됩니다.
  • AWS Bedrock / Google Vertex AI 사용자: 별도 추가 설정이 필요합니다. 각 플랫폼 공식 문서를 참고하세요.
  • 실습 프로젝트는 필수가 아닙니다. 자신의 코드베이스로 따라해도 됩니다.
  • Anthropic API 키는 선택사항입니다 (실습 기준). API 키 없이도 앱이 정적 가짜 코드를 생성합니다.

Claude Code 설치 방법

macOS (권장)
Homebrew
brew install --cask claude-code
macOS / Linux / WSL
curl 스크립트
curl -fsSL https://claude.ai/install.sh | bash
Windows CMD
CMD 스크립트
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd
첫 실행
인증 필요
터미널에서 claude 실행 → 자동 인증 프롬프트

실습 프로젝트 설정 (uigen)

  • Node.js 설치 확인 (nodejs.org)
  • uigen.zip 다운로드 후 압축 해제
  • 프로젝트 디렉토리에서 npm run setup 실행
  • (선택) .env 파일에 Anthropic API 키 설정
  • npm run dev로 앱 시작
💡 팁
자신의 기존 프로젝트를 사용해도 좋습니다. 익숙한 프로젝트에서 시작하면 학습 효과가 더 클 수 있습니다.

LESSON 03

컨텍스트 추가하기 — /init, CLAUDE.md, @ 파일 멘션

⏱ 예상 소요 시간: 25분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • /init 명령어로 CLAUDE.md를 자동 생성하고 역할을 설명할 수 있다
  • CLAUDE.md의 세 가지 위치(프로젝트·개인·전역)와 각각의 용도를 구분할 수 있다
  • # 명령어로 CLAUDE.md에 지시사항을 추가하는 방법을 실습할 수 있다
  • @ 파일 멘션으로 특정 파일을 컨텍스트에 포함하는 방법을 사용할 수 있다
핵심 요약
  • /init은 프로젝트 분석의 시작점입니다. Claude가 전체 코드베이스를 분석해 목적·명령어·패턴을 이해하고 CLAUDE.md에 요약합니다.
  • CLAUDE.md는 Claude의 영구 시스템 프롬프트입니다. 모든 요청에 자동으로 포함되어 항상 올바른 컨텍스트로 작업합니다.
  • 세 가지 위치: 프로젝트 루트(팀 공유) / CLAUDE.local.md(개인 비공유) / ~/.claude/CLAUDE.md(전역)
  • # 명령어 = 지능적 업데이트. 자연어로 CLAUDE.md를 수정하면 Claude가 기존 내용에 자연스럽게 병합합니다.
  • @ 파일 멘션으로 즉각 컨텍스트 추가: @auth처럼 입력하면 관련 파일 목록이 표시되고, 선택한 파일이 요청에 포함됩니다.

왜 컨텍스트 관리가 중요한가?

프로젝트에 수십~수백 개의 파일이 있더라도 Claude는 현재 작업과 관련된 정보만 필요합니다. 관련 없는 컨텍스트가 너무 많으면 Claude의 성능이 오히려 떨어집니다.

/init — 프로젝트 분석 및 CLAUDE.md 생성

새 프로젝트에서 처음 Claude를 시작할 때 /init을 실행하세요. Claude가 전체 코드베이스를 분석해 다음을 파악합니다:

  • 프로젝트의 목적과 아키텍처
  • 중요 명령어와 핵심 파일들
  • 코딩 패턴과 디렉토리 구조

Enter로 각 쓰기를 승인하거나, Shift+Tab으로 세션 동안 자유롭게 쓸 수 있도록 허용합니다.

CLAUDE.md 파일 위치 비교

위치파일명공유 여부용도
프로젝트 루트CLAUDE.md✅ 팀 전체아키텍처, 코딩 스타일, 중요 명령어
프로젝트 루트CLAUDE.local.md❌ 비공유개인 취향, 로컬 환경 설정
홈 디렉토리~/.claude/CLAUDE.md개인 전역모든 프로젝트 공통 규칙

# 명령어와 @ 파일 멘션

# 메모리 모드로 지시사항 추가:
# 복잡한 코드에만 주석 달기. 간단한 코드엔 주석 불필요.

# 특정 파일 컨텍스트 포함:
인증 시스템이 어떻게 동작해? @auth

# CLAUDE.md에서 파일 참조 (매번 검색 불필요):
데이터베이스 스키마는 @prisma/schema.prisma 파일에 정의됨.
생각해 볼 것들
  • 내 프로젝트에서 CLAUDE.md에 반드시 포함할 정보는 무엇인가?
  • 팀 공유 CLAUDE.md와 개인 CLAUDE.local.md를 어떻게 구분해 관리할까?
  • 자주 참조하는 파일을 CLAUDE.md에 등록하면 생산성이 얼마나 향상될까?
LESSON 04

변경사항 만들기 — 스크린샷, Planning Mode, Thinking

⏱ 예상 소요 시간: 25분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 스크린샷을 활용해 UI 변경사항을 Claude에게 정확하게 전달할 수 있다
  • Planning Mode를 활성화하고 복잡한 작업의 구현 계획을 검토할 수 있다
  • 5가지 Thinking Mode의 차이를 이해하고 상황에 맞게 선택할 수 있다
  • Planning Mode와 Thinking Mode를 언제 조합할지 판단할 수 있다
핵심 요약
  • 스크린샷은 가장 정확한 소통 도구입니다. macOS에서 Ctrl+V(Cmd+V 아님)로 붙여넣기 합니다.
  • Planning Mode는 복잡한 작업의 안전망입니다. Shift+Tab 두 번으로 활성화. Claude가 실행 전 상세 계획을 보여주고 승인을 기다립니다.
  • Thinking Mode는 추론 깊이를 조절합니다. "think" → "ultrathink" 순으로 점점 더 많은 토큰 할당.
  • Planning vs Thinking: Planning은 코드베이스 전반 이해 필요 시, Thinking은 복잡한 로직·알고리즘 문제 시.
  • 둘 다 토큰을 소비합니다. 필요한 경우에만 사용하세요.

Planning Mode vs Thinking Mode

기능활성화 방법적합한 경우
Planning ModeShift+Tab 두 번여러 파일 수정, 기능 추가, 리팩토링
think프롬프트에 "think" 입력일반적인 복잡한 문제
think more프롬프트에 "think more"중간 수준 심화 분석
ultrathink프롬프트에 "ultrathink"최대 추론 — 복잡한 알고리즘/아키텍처
⚠️ macOS 주의
스크린샷 붙여넣기는 Ctrl+V입니다. 일반 macOS 단축키 Cmd+V가 아닙니다.

  • Thinking Mode를 남용하면 어떤 비용 문제가 생길 수 있을까?
  • 스크린샷 기반 소통이 특히 유용한 상황은 어떤 경우인가?
LESSON 05

컨텍스트 관리 — Escape, /compact, /clear

⏱ 예상 소요 시간: 20분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • Escape 키로 Claude를 중단하고 올바른 방향으로 재지시할 수 있다
  • Escape 두 번으로 대화를 되감아 이전 지점으로 돌아갈 수 있다
  • /compact와 /clear의 차이를 이해하고 상황에 맞게 사용할 수 있다
  • 반복되는 실수를 # 명령어와 Escape 조합으로 영구 수정할 수 있다
핵심 요약
  • Escape = 즉각적인 중단. Claude가 잘못된 방향으로 가거나 너무 많은 것을 한번에 하려 할 때 멈추고 재지시합니다.
  • Escape + # = 반복 실수 영구 수정. Escape 중단 → # 명령으로 CLAUDE.md에 올바른 방법 기록 → 다시는 같은 실수 안 함.
  • Escape 두 번 = 대화 되감기. 불필요한 논의는 제거하되, Claude가 프로젝트에 대해 쌓은 지식은 보존.
  • /compact = 핵심 보존 압축. 관련 작업 간 전환 시 적합.
  • /clear = 완전 초기화. 완전히 다른 작업 전환 시 사용.

컨텍스트 관리 기법 비교

기법동작언제 사용컨텍스트 보존
Escape현재 응답 즉시 중단잘못된 방향 감지전체 보존
Escape ×2이전 메시지로 되감기불필요한 논의 제거선택적 보존
/compact대화 요약 압축관련 작업 전환핵심 정보 보존
/clear대화 전체 삭제완전 다른 작업 시작없음
💡 /compact vs /clear 선택 기준
Claude가 유용한 코드베이스 지식을 쌓았다면 → /compact
완전히 다른 프로젝트나 작업이라면 → /clear

훅(Hooks) 소개 — 도구 실행 전후 제어

⏱ 예상 소요 시간: 25분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • PreToolUse와 PostToolUse 훅의 차이와 활용 시나리오를 설명할 수 있다
  • 훅 설정 파일의 구조(matcher, command)를 작성할 수 있다
  • 훅을 세 가지 설정 파일 위치 중 적절한 곳에 배치할 수 있다
핵심 요약
  • 훅은 Claude의 도구 실행 흐름에 끼어드는 코드입니다. 도구 실행 전(PreToolUse) 또는 후(PostToolUse)에 사용자 정의 명령을 실행합니다.
  • PreToolUse: 도구 실행을 막을 수 있습니다. .env 파일 읽기 차단 등.
  • PostToolUse: 실행 결과에 반응합니다. 파일 수정 후 자동 포맷터 실행 등.
  • matcher로 특정 도구 대상 지정: "Write|Edit|MultiEdit"처럼 파이프로 OR 조건.
  • 절대 경로 사용 권장: 보안 공격(경로 인터셉션) 방지.

훅 설정 구조

{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Read",
      "hooks": [{
        "type": "command",
        "command": "node /absolute/path/read_hook.js"
      }]
    }],
    "PostToolUse": [{
      "matcher": "Write|Edit|MultiEdit",
      "hooks": [{
        "type": "command",
        "command": "node /absolute/path/format_hook.js"
      }]
    }]
  }
}

훅 구현하기 — .env 파일 보호 실전 예제

⏱ 예상 소요 시간: 30분
📺 원본 영상 →
이 레슨을 마치면 할 수 있는 것
  • 훅 스크립트가 stdin으로 받는 JSON 데이터 구조를 이해할 수 있다
  • exit code(0, 2)로 도구 실행을 허용/차단하는 방법을 구현할 수 있다
  • .env 파일 접근을 차단하는 실제 훅을 작성하고 테스트할 수 있다
  • $PWD 플레이스홀더 패턴으로 팀 공유와 절대 경로를 동시에 해결할 수 있다
핵심 요약
  • 훅 스크립트는 stdin으로 JSON을 받습니다. session_id, hook_event_name, tool_name, tool_input 등 포함.
  • exit code로 결과 전달: 0 = 허용, 2 = 차단(PreToolUse만). 차단 시 stderr 메시지가 Claude에게 피드백.
  • Read|Grep 모두 차단 필요: .env 보호 시 Grep으로 우회하는 것도 막아야 합니다.
  • $PWD 플레이스홀더: settings.example.json에 $PWD 사용 → npm run setup 시 실제 경로로 치환 → settings.local.json 생성.

.env 파일 보호 훅 구현

// hooks/read_hook.js
async function main() {
  const chunks = [];
  for await (const chunk of process.stdin) chunks.push(chunk);

  const toolArgs = JSON.parse(Buffer.concat(chunks).toString());
  const readPath = toolArgs.tool_input?.file_path || "";

  if (readPath.includes('.env')) {
    console.error("보안 정책: .env 파일은 읽을 수 없습니다");
    process.exit(2);  // 2 = 차단
  }
  // exit 0 = 허용 (암묵적)
}
main();

디버깅: stdin 데이터 구조 확인

  }]
