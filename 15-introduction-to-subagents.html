레슨 1 · What are subagents?

서브에이전트란?

📺 원본 영상 →

🎯 학습 목표

  • 서브에이전트가 무엇이며 Claude Code에서 어떻게 작동하는지 설명할 수 있다
  • 서브에이전트가 컨텍스트 창 관리에 어떻게 도움이 되는지 이해한다
  • 내장 서브에이전트와 커스텀 서브에이전트의 차이를 구분할 수 있다

💡 핵심 요약

  • → 서브에이전트 = Claude Code가 작업을 위임하는 전문화된 보조자. 별도의 컨텍스트 창에서 실행
  • → 메인 컨텍스트가 깨끗하게 유지됨 — 중간 단계(파일 읽기·검색·도구 호출)가 서브에이전트 안에 격리
  • → 서브에이전트는 작업을 마치면 요약만 반환 — 전체 대화는 폐기됨
  • → 내장 서브에이전트: General purpose, Explore, Plan. 커스텀 서브에이전트도 직접 생성 가능

서브에이전트의 작동 원리

Claude Code와 대화할 때마다 메인 컨텍스트 창에 내용이 쌓입니다. 모든 도구 호출, 파일 읽기, 검색 결과가 저장되죠. 이 공간은 유한합니다.

서브에이전트는 별도의 컨텍스트 창을 열어 이 문제를 해결합니다. 서브에이전트는 두 가지를 받습니다:

  • 서브에이전트의 역할과 동작을 정의하는 커스텀 시스템 프롬프트
  • 부모 에이전트가 요청을 바탕으로 작성한 작업 설명

서브에이전트는 독자적으로 작업합니다 — 파일을 읽고, 검색을 실행하고, 코드를 편집합니다. 완료되면 요약만 메인 대화로 돌아옵니다. 서브에이전트 대화 전체는 폐기됩니다.

실전 예시

낯선 코드베이스에서 환불을 처리하는 서비스를 찾아야 한다고 가정합니다.

서브에이전트 없이

15개 파일 읽기, 여러 번 검색, 다중 함수 호출 추적 — 필요한 사실 하나를 위해 모든 과정이 컨텍스트에 쌓임

서브에이전트 사용

Explore 서브에이전트가 독자적으로 탐색하고 집중된 답변만 반환. 메인 컨텍스트에는 질문과 요약만 기록

내장 서브에이전트

🤖

General purpose

탐색과 실행이 모두 필요한 다단계 작업용 범용 서브에이전트

🔍

Explore

코드베이스의 빠른 검색 및 탐색 전문

📋

Plan

구현 계획 수립 전 코드베이스 리서치 및 분석에 사용

내장 서브에이전트 외에도 커스텀 시스템 프롬프트와 도구 접근 권한을 갖춘 자신만의 서브에이전트를 만들 수 있습니다.

서브에이전트의 핵심 이점

  • 작업을 집중된 조각으로 분리 — 각 서브에이전트가 특정 작업에 집중
  • 모든 중간 작업을 격리해 메인 컨텍스트 창을 깨끗하게 유지
  • 필요한 정보만 간결한 요약으로 반환
⚠️

트레이드오프

서브에이전트가 어떻게 결론에 도달했는지 가시성이 사라집니다. 과정보다 결과가 중요한 작업에 적합합니다.

레슨 2 · Creating a subagent

서브에이전트 만들기

📺 원본 영상 →

🎯 학습 목표

  • /agents 명령을 사용해 서브에이전트를 생성할 수 있다
  • 서브에이전트의 범위, 도구, 모델, 색상을 설정할 수 있다
  • 설정 파일의 각 필드(name, description, tools, model, color)의 역할을 이해한다

💡 핵심 요약

  • → /agents 명령 → Create new agent → 범위 선택 → Claude가 설정 자동 생성
  • → 설정 파일 위치: .claude/agents/your-agent-name.md (YAML frontmatter + 시스템 프롬프트)
  • → description 필드가 핵심 — Claude가 언제 서브에이전트를 사용할지와 어떤 입력 프롬프트를 쓸지 모두 결정
  • → 'proactively'를 description에 포함하면 Claude가 자동으로 서브에이전트 위임을 제안

생성 과정

  • 1

    /agents 명령 실행

    Claude Code에서 /agents를 입력하면 서브에이전트 관리 인터페이스가 열립니다. Create new agent 선택.

  • 2

    범위 선택

    Project-level (현재 프로젝트에서만) 또는 User-level (모든 프로젝트에서 공유) 중 선택.

  • 3

    Claude가 설정 생성

    서브에이전트가 할 일을 설명하면 Claude가 이름, 설명, 시스템 프롬프트를 생성해줍니다.

  • 4

    도구 커스터마이즈

    Read-only, Edit, Execution, MCP, Other 도구 카테고리 중 실제 필요한 것만 선택.

  • 5

    모델 & 색상 선택

    Haiku(빠름), Sonnet(균형), Opus(심층), Inherit(메인과 동일) 중 선택. 색상은 UI에서 서브에이전트 구별용.

설정 파일 구조

생성 완료 후 .claude/agents/your-agent-name.md에 저장됩니다:

---
name: code-quality-reviewer
description: Use this agent when you need to review recently written or modified code for quality, security, and best practice compliance.
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch
model: sonnet
color: purple
---

You are an expert code reviewer specializing in quality assurance, security best practices, and adherence to project standards...
필드역할
name고유 식별자. @agent name으로 직접 호출 가능
descriptionClaude가 언제 서브에이전트를 쓸지와 어떤 입력을 전달할지 결정 (단일 라인)
tools접근 가능한 도구 목록 — 언제든 편집 가능
modelsonnet / opus / haiku / inherit
colorUI 구별용 색상

자동 위임 설정

description에 "proactively"를 포함하면 Claude가 명시적 요청 없이도 자동으로 서브에이전트에 위임을 제안합니다:

description: Proactively suggest running this agent after major code changes to ensure quality and security standards are met.

구체적인 예시 대화를 description에 추가할수록 Claude가 언제 위임해야 하는지 더 잘 이해합니다.

레슨 3 · Designing effective subagents

효과적인 서브에이전트 설계

📺 원본 영상 →

🎯 학습 목표

  • 입력 프롬프트를 형성하는 description을 작성할 수 있다
  • 서브에이전트에 구조화된 출력 형식을 정의할 수 있다
  • 도구 접근을 적절히 제한하는 방법을 이해한다

💡 핵심 요약

  • → description은 두 가지 역할: 서브에이전트 실행 시점 결정 + 메인 에이전트가 쓸 입력 프롬프트 형성
  • → 출력 형식 정의가 가장 중요한 개선 — 자연스러운 종료 지점 + 과도한 실행 방지
  • → Obstacles Encountered 섹션을 출력 형식에 추가 — 발견한 해결책이 메인 스레드로 전달됨
  • → 도구는 실제 필요한 것만: 리서치=읽기 전용, 코드 리뷰어=Bash+읽기, 코드 수정=Edit+Write

Description이 입력 프롬프트를 형성하는 방식

메인 에이전트에 메시지를 보낼 때, 사용 가능한 모든 서브에이전트의 name과 description이 시스템 프롬프트에 포함됩니다. 이것이 메인 에이전트가 어떤 서브에이전트를 언제 실행할지 결정하는 방법입니다.

description은 두 번째 역할도 합니다: 메인 에이전트가 서브에이전트를 실행할 때 작업을 시작하는 입력 프롬프트를 작성하는 데 description을 가이드로 사용합니다.

일반적인 description

메인 에이전트 입력: "git diff로 현재 변경사항 찾아봐" → 막연함

구체적인 description

"검토할 파일을 정확히 명시할 것" 추가 → 특정 파일 목록이 있는 구체적인 입력 생성

출력 형식 정의

서브에이전트에 가장 중요한 개선사항은 시스템 프롬프트에 출력 형식을 정의하는 것입니다:

  • 자연스러운 종료 지점 생성 — 각 섹션을 채웠을 때 완료를 인식
  • 과도한 실행 방지 — 정의 없이는 언제 리서치를 멈춰야 할지 모름
Provide your review in a structured format:

1. Summary: Brief overview of what you reviewed and overall assessment
2. Critical Issues: Security vulnerabilities, data integrity risks, or logic errors
3. Major Issues: Quality problems, architecture misalignment
4. Minor Issues: Style inconsistencies, documentation gaps
5. Recommendations: Suggestions for improvement
6. Approval Status: Ready to merge or requires changes
7. Obstacles Encountered: Setup issues, workarounds, commands with special flags

장애물 보고

서브에이전트가 작업 중 해결책을 발견했을 때 — 종속성 문제 해결, 특수 플래그가 필요한 명령 등 — 그 세부 사항이 반환하는 요약에 나타나야 합니다. 그렇지 않으면 메인 스레드가 같은 해결책을 다시 발견해야 합니다.

출력 형식에 Obstacles Encountered 섹션을 추가하면 이 정보가 안정적으로 표면화됩니다.

도구 접근 제한

서브에이전트 유형필요한 도구이유
리서치/읽기 전용Glob, Grep, Read파일을 실수로 수정하지 않음
코드 리뷰어Bash (git diff), Read변경 내용 확인은 하되 Edit/Write 불필요
스타일링/코드 수정Edit, Write 포함실제로 코드를 변경하는 것이 역할
레슨 4 · Using subagents effectively

서브에이전트 효과적으로 활용

📺 원본 영상 →

🎯 학습 목표

  • 서브에이전트가 실제로 도움이 되는 상황과 방해가 되는 상황을 구분할 수 있다
  • 리서치, 코드 리뷰, 커스텀 시스템 프롬프트에 서브에이전트를 활용할 수 있다
  • 안티패턴(전문가 주장, 순차 파이프라인, 테스트 실행자)을 피할 수 있다

💡 핵심 요약

  • → 핵심 결정 규칙: 중간 작업이 중요한가? 아니라면 위임. 중요하다면 메인 스레드에 유지
  • → 서브에이전트 적합: 리서치/탐색, 코드 리뷰, 커스텀 시스템 프롬프트가 필요한 작업
  • → 서브에이전트 부적합: '전문가' 페르소나, 순차 의존 파이프라인, 전체 출력이 필요한 테스트 실행
  • → 코드 리뷰어 서브에이전트: 작성에 관여하지 않은 새로운 컨텍스트에서 검토 → 더 객관적

서브에이전트가 빛나는 경우

서브에이전트는 탐색이 실행과 분리될 때 가장 잘 작동합니다. 작업의 각 단계가 이전 단계에서 발견한 것에 의존한다면 메인 스레드에 유지하세요. 결과만 필요하고 과정은 신경 쓰지 않는다면 위임하세요.

🔍

리서치 작업

서브에이전트가 수십 개의 파일을 읽고 함수 호출을 추적. 메인 스레드는 깨끗한 요약만 받음.

👀

코드 리뷰

새로운 컨텍스트에서 변경 사항을 검토. 작성 과정을 모르는 외부 시각으로 더 날카로운 피드백.

🎯

커스텀 시스템 프롬프트

기본 Claude Code 프롬프트와 다른 지침이 필요한 경우 (카피라이팅, 스타일링 등).

코드 리뷰 심화

Claude는 다른 사람이 작성한 코드로 제시될 때 더 효과적으로 리뷰합니다. 여러 턴에 걸쳐 메인 스레드에서 기능을 개발했다면, 같은 스레드에 리뷰를 요청하면 약한 피드백이 나옵니다 — Claude가 코드 생성에 관여했기 때문입니다.

리뷰어 서브에이전트는 별도 컨텍스트에서 변경 사항을 봅니다. git diff를 실행하고 수정된 파일을 읽으며 작성 기록 없이 전문화된 리뷰 기준을 적용합니다.

피해야 할 안티패턴

⚠️

전문가 주장 (Expert Claims)

"당신은 Python 전문가입니다" 같은 프롬프트는 도움이 안 됩니다. Claude는 이미 그 지식을 가지고 있습니다. 소위 전문가 서브에이전트가 메인 스레드에서 직접 할 수 없는 것은 없습니다.

⚠️

순차 파이프라인 (Sequential Pipelines)

버그 재현 → 디버그 → 수정의 3에이전트 파이프라인: 각 단계가 이전 단계의 발견에 의존하므로 핸드오프에서 정보가 손실됩니다. 버그 수정은 거의 항상 이런 구조입니다.

⚠️

테스트 실행자 (Test Runners)

테스트가 실패할 때 전체 출력이 필요합니다. "테스트 실패"만 반환하는 서브에이전트는 직접 출력에서 볼 수 있었던 세부 정보를 얻기 위해 추가 디버그 스크립트가 필요합니다.

의사결정 규칙

서브에이전트 사용메인 스레드에 유지
리서치와 탐색"전문가" 페르소나
코드 리뷰각 단계가 이전에 의존하는 다단계 파이프라인
커스텀 시스템 프롬프트가 필요한 작업전체 출력이 필요한 테스트 실행

핵심 질문

  • 중간 작업이 중요한가? 아니라면 서브에이전트에게 위임하세요.
  • 결과만 필요하고 과정은 신경 쓰지 않는가? 서브에이전트가 적합합니다.
  • 각 단계가 이전 발견에 의존하는가? 메인 스레드에 유지하세요.
