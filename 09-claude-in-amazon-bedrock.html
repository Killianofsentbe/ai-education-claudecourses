LESSON 01

Amazon Bedrock & Claude 모델 이해

AWS 에코시스템에서 Claude를 활용하는 방법과 Bedrock의 고유한 특성을 이해합니다

이 레슨에서 배우는 것

  • Amazon Bedrock이 무엇이며 일반 Anthropic API와 어떻게 다른지
  • Bedrock에서의 Claude 모델 ID 형식 및 추론 프로파일
  • boto3를 사용한 Bedrock 클라이언트 설정 및 인증
  • AWS 리전별 모델 가용성 문제와 해결 방법

Amazon Bedrock이란?

Amazon Bedrock은 AWS에서 제공하는 완전 관리형 AI 서비스로, Claude를 포함한 여러 AI 모델에 단일 API를 통해 접근할 수 있습니다. 기존 AWS 인프라와 통합이 매끄럽고, AWS의 보안·규정 준수 체계를 그대로 활용할 수 있습니다.

일반 Anthropic API

Anthropic SDK 사용, ANTHROPIC_API_KEY로 인증, 직접 모델에 접근. AWS 인프라와 별도로 운영됩니다.

Amazon Bedrock

boto3 SDK 사용, AWS IAM으로 인증, AWS 생태계 내 통합. 기존 AWS 워크로드와 자연스럽게 결합됩니다.

Bedrock 클라이언트 설정

import boto3

# Bedrock Runtime 클라이언트 생성
# AWS 인증 정보은 환경 변수 또는 IAM 역할로 자동 처리
client = boto3.client(
    "bedrock-runtime",
    region_name="us-west-2"  # 리전 지정 필수
)
⚠️ AWS 인증 정보 필요

boto3는 AWS 인증 정보이 있어야 합니다. 로컬 개발 시 ~/.aws/credentials를 설정하거나, EC2/Lambda에서는 IAM 역할을 사용하세요. AWS_ACCESS_KEY_ID와 AWS_SECRET_ACCESS_KEY 환경 변수도 사용 가능합니다.

모델 ID와 추론 프로파일

Bedrock에서 Claude를 사용할 때 모델 가용성 문제가 생길 수 있습니다. 특정 모델이 특정 리전에 없을 수 있습니다. 추론 프로파일(Inference Profile)로 이 문제를 해결할 수 있습니다.

# ❌ 일반 모델 ID (리전 문제 발생 가능)
model_id = "anthropic.claude-sonnet-4-0-20250514-v1:0"

# ✅ 추론 프로파일 (크로스 리전 자동 라우팅)
model_id = "us.anthropic.claude-sonnet-4-0-20250514-v1:0"
# 접두사 'us.' 또는 'eu.' 등으로 리전 그룹 지정

# AWS Bedrock 콘솔 > Cross-region inference 에서 확인
모델Bedrock 추론 프로파일 ID 예시특성
Claude Opusus.anthropic.claude-opus-4-5-...최고 지능, 복잡한 추론
Claude Sonnetus.anthropic.claude-sonnet-4-5-...균형 (대부분의 프로덕션)
Claude Haikuus.anthropic.claude-haiku-4-5-...최고 속도·비용 효율

다중 모델 전략

실제 프로덕션 시스템에서는 하나의 모델만 사용하지 않습니다. 각 단계에 최적의 모델을 선택하세요:

Haiku 적합 사례

실시간 사용자 채팅, 간단한 분류, 대용량 데이터 처리, 테스트 데이터셋 생성

Sonnet 적합 사례

핵심 비즈니스 로직, 코드 생성/수정, RAG 응답 생성, 대부분의 프로덕션 작업

Opus 적합 사례

복잡한 분석, 전략적 의사결정, 고급 추론이 필요한 연구 작업

비용 최적화 팁

Haiku(저렴) → 결과 검토 → Sonnet(보통) → 복잡한 케이스만 Opus(고가)로 에스컬레이션

핵심 요약

  • Amazon Bedrock은 AWS 생태계에서 Claude를 완전 관리형으로 활용하는 서비스입니다
  • boto3 bedrock-runtime 클라이언트를 생성하고 AWS IAM으로 인증합니다
  • 추론 프로파일 ID(예: us.anthropic...)를 사용하면 리전 문제를 자동으로 해결합니다
  • Haiku/Sonnet/Opus를 용도에 맞게 조합하면 품질과 비용을 최적화할 수 있습니다
  • AWS 콘솔 > Bedrock > Cross-region inference에서 추론 프로파일 ID를 확인하세요

  • 추론 프로파일의 크로스 리전 라우팅이 지연 시간에 미치는 영향은?
  • IAM 역할 기반 인증이 API 키 방식보다 어떤 보안적 장점을 제공하는가?
LESSON 02

Bedrock API & 대화 관리

client.converse()로 첫 요청을 만들고, 멀티턴 대화를 Bedrock API로 구현합니다

이 레슨에서 배우는 것

  • Bedrock의 converse() 메서드와 메시지 구조
  • Anthropic API와 다른 응답 구조에서 텍스트 추출
  • 멀티턴 대화를 위한 히스토리 관리
  • 스트리밍 응답 (converse_stream()) 구현

첫 번째 Bedrock 요청

import boto3

client = boto3.client("bedrock-runtime", region_name="us-west-2")
model_id = "us.anthropic.claude-sonnet-4-5-20251001-v1:0"

# 메시지 구조 (Bedrock 특유의 형식)
user_message = {
    "role": "user",
    "content": [
        {"text": "What is quantum computing? Answer in one sentence."}
    ]
}

# converse() 메서드로 요청
response = client.converse(
    modelId=model_id,
    messages=[user_message]
)

# 응답 텍스트 추출 (Bedrock 응답 구조)
text = response["output"]["message"]["content"][0]["text"]
print(text)
⚠️ Anthropic API와 다른 응답 구조

Anthropic API: message.content[0].text
Amazon Bedrock: response["output"]["message"]["content"][0]["text"]

멀티턴 대화 관리

Claude는 Bedrock에서도 무상태(stateless)입니다. 모든 대화 히스토리를 매 요청에 포함해야 합니다.

def add_user_message(messages, text):
    messages.append({"role": "user", "content": [{"text": text}]})

def add_assistant_message(messages, text):
    messages.append({"role": "assistant", "content": [{"text": text}]})

def chat(messages, system=None):
    params = {
        "modelId": model_id,
        "messages": messages,
        "inferenceConfig": {"maxTokens": 1000}
    }
    if system:
        params["system"] = [{"text": system}]

    response = client.converse(**params)
    return response["output"]["message"]["content"][0]["text"]

# 멀티턴 대화 예시
messages = []
add_user_message(messages, "Define quantum computing in one sentence")
answer = chat(messages)

add_assistant_message(messages, answer)  # 히스토리에 추가!
add_user_message(messages, "Give me a real-world application")
final = chat(messages)

스트리밍 응답

def chat_stream(messages, system=None):
    params = {
        "modelId": model_id,
        "messages": messages,
        "inferenceConfig": {"maxTokens": 1000}
    }
    if system:
        params["system"] = [{"text": system}]

    response = client.converse_stream(**params)

    full_text = ""
    for event in response["stream"]:
        if "contentBlockDelta" in event:
            delta = event["contentBlockDelta"]["delta"]
            if "text" in delta:
                chunk = delta["text"]
                print(chunk, end="", flush=True)
                full_text += chunk

    return full_text

Anthropic API vs Amazon Bedrock 비교

항목Anthropic APIAmazon Bedrock
SDKanthropic 패키지boto3 패키지
인증ANTHROPIC_API_KEYAWS IAM / 인증 정보

핵심 요약

  • Bedrock의 converse()는 Anthropic의 messages.create()와 동일한 역할을 합니다
  • 메시지 content는 [{"text": "..."}] 리스트 형식이어야 합니다 (멀티모달 지원을 위해)
  • Claude는 Bedrock에서도 무상태 — 매 요청에 전체 대화 히스토리를 포함해야 합니다
  • 스트리밍은 converse_stream()과 이벤트 스트림으로 구현합니다
  • 응답 구조가 Anthropic API와 다르니 주의 (response["output"]["message"]["content"][0]["text"])

LESSON 03

시스템 프롬프트 & 출력 제어

Bedrock API에서 시스템 프롬프트, Temperature, 구조화 출력을 구현합니다

이 레슨에서 배우는 것

  • Bedrock에서 시스템 프롬프트를 전달하는 방법 (별도 파라미터)
  • inferenceConfig로 Temperature, maxTokens 등 조절
  • 어시스턴트 프리필 + 중단 시퀀스로 구조화 출력 생성
  • 스트리밍과 시스템 프롬프트 조합

시스템 프롬프트 (Bedrock 방식)

Bedrock에서 시스템 프롬프트는 system 파라미터로 별도 전달합니다. 구조가 메시지와 동일합니다.

system_prompt = """
You are a patient math tutor.
Do not directly answer a student's questions.
Guide them to a solution step by step.
"""

response = client.converse(
    modelId=model_id,
    messages=messages,
    system=[{"text": system_prompt}],  # 리스트로 감싸야 함!
    inferenceConfig={"maxTokens": 1000}
)
💡 Bedrock 시스템 프롬프트 특이사항

Anthropic API: system="..." (문자열)
Amazon Bedrock: system=[{"text": "..."}] (리스트 of 딕셔너리)

inferenceConfig로 출력 제어

response = client.converse(
    modelId=model_id,
    messages=messages,
    inferenceConfig={
        "maxTokens": 1000,          # 최대 생성 토큰
        "temperature": 0.7,         # 창의성 조절 (0~1)
        "topP": 0.9,                # 핵 샘플링
        "stopSequences": ["END"]    # 중단 시퀀스
    }
)
Temperature특성적합한 사례
0.0 – 0.3결정적, 일관된 출력코딩, 데이터 추출, 분류
0.4 – 0.7균형, 적당한 다양성요약, 교육 콘텐츠, Q&A
0.8 – 1.0창의적, 다양한 출력브레인스토밍, 창작, 아이디어

구조화 출력: 프리필 + 중단 시퀀스

messages = []
add_user_message(messages, "Generate an AWS IAM policy as JSON")

# 어시스턴트 메시지 프리필로 JSON 시작 강제
add_assistant_message(messages, "```json")

response = client.converse(
    modelId=model_id,
    messages=messages,
    inferenceConfig={
        "maxTokens": 1000,
        "stopSequences": ["```"]  # 닫는 코드 블록에서 중단
    }
)

raw_json = response["output"]["message"]["content"][0]["text"]
import json
data = json.loads(raw_json)  # 순수 JSON만 파싱

핵심 요약

  • Bedrock 시스템 프롬프트는 system=[{"text": "..."}] 형식입니다
  • inferenceConfig 딕셔너리로 maxTokens, temperature, stopSequences를 한번에 설정합니다
  • 프리필 + 중단 시퀀스 기법은 Bedrock에서도 동일하게 작동합니다
  • Temperature 선택: 정확성 중심(낮음) vs 창의성 중심(높음)

  • inferenceConfig의 topP와 temperature를 동시에 사용할 때 어떤 효과가 있을까?
LESSON 04

프롬프트 평가 (Evals)

Bedrock API로 체계적인 Eval 파이프라인을 구축해 프롬프트를 데이터 기반으로 개선합니다

이 레슨에서 배우는 것

  • 5단계 Eval 워크플로우의 전체 구조
  • Claude Haiku로 테스트 데이터셋 자동 생성
  • 모델 기반 채점과 코드 기반 채점 구현
  • 점수 기반 프롬프트 반복 개선

5단계 Eval 워크플로우

1
프롬프트 초안

기준 프롬프트를 작성합니다. 처음에는 단순하게 시작해도 됩니다.

2
Eval 데이터셋 생성

실제 사용 사례를 대표하는 입력 샘플을 Claude Haiku로 자동 생성합니다.

3
Claude에 통과

각 입력을 프롬프트와 결합해 Claude에게 보내고 출력을 수집합니다.

4
채점

모델 기반 또는 코드 기반 채점으로 각 출력에 점수를 매깁니다.

5
반복 개선

점수를 기준으로 프롬프트를 수정하고 전 과정을 반복합니다.

Bedrock으로 Eval 구현

import json

haiku_model = "us.anthropic.claude-haiku-4-5-20251001-v1:0"
sonnet_model = "us.anthropic.claude-sonnet-4-5-20251001-v1:0"

def generate_dataset():
    """Haiku로 저비용 테스트 데이터셋 생성"""
    prompt = """Generate 5 test cases for an AWS code generation prompt.
Each should be a task requiring Python, JSON, or Regex.

Output as JSON array:
[{"task": "..."}, ...]"""

    messages = [{"role": "user", "content": [{"text": prompt}]},
                {"role": "assistant", "content": [{"text": "```json"}]}]

    response = client.converse(
        modelId=haiku_model,
        messages=messages,
        inferenceConfig={"maxTokens": 1000, "stopSequences": ["```"]}
    )
    return json.loads(response["output"]["message"]["content"][0]["text"])

def run_prompt(test_case, prompt_template):
    """Sonnet으로 실제 프롬프트 실행"""
    prompt = prompt_template.format(task=test_case["task"])
    messages = [{"role": "user", "content": [{"text": prompt}]}]
    response = client.converse(modelId=sonnet_model, messages=messages,
                               inferenceConfig={"maxTokens": 500})
    return response["output"]["message"]["content"][0]["text"]

def score_output(task, output):
    """Claude가 출력을 채점"""
    scoring_prompt = f"""Rate this AI output on a scale of 1-10.
Task: {task}
Output: {output}
Return only a JSON: {{"score": N, "reason": "..."}}"""

    messages = [{"role": "user", "content": [{"text": scoring_prompt}]},
                {"role": "assistant", "content": [{"text": "{"}]}]

    response = client.converse(
        modelId=haiku_model,
        messages=messages,
        inferenceConfig={"maxTokens": 200, "stopSequences": ["}"]}
    )
    result = "{" + response["output"]["message"]["content"][0]["text"] + "}"
    return json.loads(result)

채점 방법 비교

방법구현장점적합 케이스
모델 채점Claude Haiku로 자동 평가유연, 복잡한 기준 적용 가능오픈 엔드 응답, 품질 평가
코드 채점정규식, JSON 파싱, 단위 테스트빠르고 무료, 완전 객관적포맷 검사, 코드 실행 결과

핵심 요약

  • Eval은 Bedrock API로도 동일하게 구현할 수 있으며, 데이터 생성에는 Haiku를 사용하세요
  • 프리필 + 중단 시퀀스로 JSON 출력을 안정적으로 파싱합니다
  • Eval 점수는 프롬프트 변경의 효과를 객관적으로 측정하는 유일한 방법입니다
  • Bedrock의 Pay-per-use 모델로 Eval 비용도 최적화할 수 있습니다

LESSON 05

프롬프트 엔지니어링

Bedrock 환경에서도 동일하게 적용되는 4가지 핵심 프롬프트 엔지니어링 기법

이 레슨에서 배우는 것

  • 명확하고 직접적인 지시로 Eval 점수 향상
  • 구체적인 가이드라인으로 출력 품질 일관성 확보
  • XML 태그로 복잡한 프롬프트 구조화
  • One-shot/Multi-shot 예시로 엣지 케이스 처리

기법 1: 명확하고 직접적으로

나쁜 예 ❌좋은 예 ✓점수 변화
"What should this person eat?""Generate a one-day meal plan for an athlete that meets their dietary restrictions."2.32 → 3.92
"Help me with AWS stuff""Write a Python Lambda function that processes S3 put events and logs metadata to CloudWatch."큰 폭 향상

from datetime import datetime

def get_current_datetime(date_format="%Y-%m-%d %H:%M:%S"):
    return datetime.now().strftime(date_format)

    messages=messages,

if stop_reason == "tool_use":
    # 도구 호출 정보 추출
    for block in output_message["content"]:
        if block.get("toolUse"):
            tool_use = block["toolUse"]
            tool_name = tool_use["name"]
            tool_input = tool_use["input"]
            tool_use_id = tool_use["toolUseId"]

            # 도구 결과를 히스토리에 추가
            messages.append(output_message)  # 어시스턴트 응답 추가
            messages.append({
                "role": "user",
                "content": [{
                    "toolResult": {
                        "toolUseId": tool_use_id,
                        "content": [{"text": str(result)}]
                    }
                }]
            })

RAG (검색 증강 생성)

AWS 생태계(S3, OpenSearch, Kendra)와 통합하는 RAG 파이프라인을 Bedrock에서 구축합니다

이 레슨에서 배우는 것

  • RAG 파이프라인의 전체 구조 (청킹 → 임베딩 → 검색 → 생성)
  • AWS 서비스를 활용한 벡터 저장 및 검색
  • 하이브리드 검색 전략 (시맨틱 + 어휘)
  • Bedrock Knowledge Bases 서비스 소개

RAG 전체 흐름

1
문서 수집 & 청킹

S3에 저장된 문서를 읽어 1000자 단위 청크로 분할합니다. 200자 오버랩으로 경계 문맥을 보존합니다.

2
임베딩 생성

VoyageAI API로 각 청크를 벡터로 변환합니다. Anthropic은 자체 임베딩 서비스를 제공하지 않습니다.

3
벡터 DB 저장

OpenSearch Service나 pgvector(RDS)에 벡터를 저장합니다. Amazon Kendra도 관리형 옵션입니다.

4
쿼리 처리

사용자 질문을 임베딩으로 변환하고 유사한 청크를 검색합니다.

5
Claude로 응답 생성

관련 청크 + 질문을 Bedrock converse()로 전달해 최종 응답을 생성합니다.

AWS 기반 RAG 구현

import voyageai
import boto3
import json

# 임베딩 클라이언트
voyage_client = voyageai.Client()

def generate_embedding(text):
    result = voyage_client.embed([text], model="voyage-3-large", input_type="document")
    return result.embeddings[0]

def semantic_search(query, chunks_with_embeddings, top_k=3):
    import numpy as np
    query_emb = voyage_client.embed([query], model="voyage-3-large", input_type="query").embeddings[0]

    scores = []
    for chunk, emb in chunks_with_embeddings:
        score = np.dot(query_emb, emb)
        scores.append((score, chunk))

    return [chunk for _, chunk in sorted(scores, reverse=True)[:top_k]]

def rag_query(user_question, chunks_with_embeddings):
    relevant_chunks = semantic_search(user_question, chunks_with_embeddings)
    context = "\n\n".join(relevant_chunks)

    prompt = f"""Answer the question based on the provided context.


{context}


{user_question}

If the answer is not in the context, say so clearly."""

    messages = [{"role": "user", "content": [{"text": prompt}]}]
    response = bedrock_client.converse(
        modelId=model_id,
        messages=messages,
        inferenceConfig={"maxTokens": 1000}
    )
    return response["output"]["message"]["content"][0]["text"]

AWS 관리형 옵션: Bedrock Knowledge Bases

직접 구현 대신 AWS Bedrock Knowledge Bases를 사용하면 RAG 파이프라인을 완전 관리형으로 실행할 수 있습니다.

직접 구현 RAG

완전한 제어, 커스터마이징 가능, 더 복잡한 구현 필요. 특수한 청킹/검색 전략이 필요할 때 적합합니다.

Bedrock Knowledge Bases

완전 관리형, S3와 자동 통합, 간단한 API. 빠른 구축이 필요하거나 표준 RAG로 충분한 경우 적합합니다.

핵심 요약

  • RAG = 문서 청킹 → 임베딩(VoyageAI) → 벡터 DB → 쿼리 검색 → Bedrock 응답 생성
  • AWS 환경에서는 S3(저장), OpenSearch(벡터 DB), Bedrock(생성)을 자연스럽게 연결합니다
  • Bedrock Knowledge Bases로 RAG를 완전 관리형으로 운영할 수 있습니다
  • 하이브리드 검색(시맨틱 + BM25)이 단독 시맨틱 검색보다 더 나은 결과를 줍니다

Bedrock 기반 MCP 클라이언트

from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

class BedrockMCPClient:
    def __init__(self, model_id, region="us-west-2"):
        self.bedrock = boto3.client("bedrock-runtime", region_name=region)
        self.model_id = model_id
        self._session = None

    async def connect(self, server_command):
        params = StdioServerParameters(command=server_command[0], args=server_command[1:])
        stdio, write = await stdio_client(params).__aenter__()
        self._session = ClientSession(stdio, write)
        await self._session.initialize()

    async def list_tools(self):
        result = await self._session.list_tools()
        # MCP 도구를 Bedrock toolSpec 형식으로 변환
        return [{
            "toolSpec": {
                "name": tool.name,
                "description": tool.description,
                "inputSchema": {"json": tool.inputSchema}
            }
        } for tool in result.tools]

    async def call_tool(self, name, arguments):
        result = await self._session.call_tool(name, arguments)
        return result.content[0].text if result.content else ""

            messages=messages,
            toolConfig={"tools": tools},
            inferenceConfig={"maxTokens": 2000}
        )

        output_message = response["output"]["message"]
        stop_reason = response["stopReason"]

        # 작업 완료
        if stop_reason == "end_turn":
            return output_message["content"][0]["text"]

        # 도구 실행
        if stop_reason == "tool_use":
            messages.append(output_message)
            tool_results = []

            for block in output_message["content"]:
                if "toolUse" in block:
                    tool_name = block["toolUse"]["name"]
                    tool_input = block["toolUse"]["input"]

                    # CloudWatch에 로깅
                    log_tool_call(tool_name, tool_input)

                    result = await mcp_client.call_tool(tool_name, tool_input)
                    tool_results.append({
                        "toolResult": {
                            "toolUseId": block["toolUse"]["toolUseId"],
                            "content": [{"text": str(result)}]
                        }
                    })

            messages.append({"role": "user", "content": tool_results})

    return "최대 반복 횟수 초과"

Bedrock Agents 관리형 서비스

직접 에이전트 루프를 구현하는 것 외에, AWS는 Bedrock Agents라는 완전 관리형 에이전트 서비스를 제공합니다.

직접 구현 에이전트

converse() + toolConfig로 구현. 완전한 제어권, 커스터마이징 가능. 복잡한 워크플로우에 적합합니다.

Bedrock Agents

AWS 완전 관리형. Lambda Action Groups으로 도구 연결. 빠른 프로토타이핑과 표준 에이전트 패턴에 적합합니다.

핵심 요약

  • AWS 에이전트는 IAM 최소 권한 + CloudWatch 로깅 + SNS 알림의 3요소로 안전하게 설계하세요
  • 에이전트 루프: converse() → tool_use 감지 → 도구 실행 → 결과 전송 → 반복
  • 최대 반복 횟수를 설정해 무한 루프를 방지하세요
  • Bedrock Agents는 완전 관리형으로 빠른 구축이 필요할 때 활용합니다
  • 이 코스에서 배운 boto3, Tool Use, RAG, MCP 모두가 AWS 에이전트의 핵심 구성 요소입니다
